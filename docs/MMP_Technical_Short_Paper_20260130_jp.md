## 概要

AIエージェントが多様なプラットフォームやフレームワークで普及するにつれ、標準化されたエージェント間通信の必要性が重要になっている。Anthropic の Model Context Protocol (MCP) は LLM とツール間の通信を、Google の Agent2Agent Protocol (A2A) はエージェント間の相互運用性を実現しているが、いずれもプロトコル機能を静的な定義として扱っている。本論文では、Model Meeting Protocol (MMP) を提案する。これは「Protocol as Skill」という新しいパラダイムを実装したエージェント間通信のオープン標準であり、プロトコル定義自体が交換可能なスキルとしてエージェント間の相互作用を通じて進化できる。MMP は KairosChain フレームワークのスキル管理を分散環境に拡張し、エージェントが互いを発見し、機能を交渉し、レイヤー化された制約と人間承認メカニズムによる安全性を維持しながら通信パターンを共進化させることを可能にする。このアプローチが、AIエージェントが信頼ネットワーク内で特化した通信プロトコルを自律的に開発・伝播する未来の基盤を提供すると主張する。

**キーワード:** エージェント間通信、プロトコル進化、MCP、A2A、スキル交換、共進化、分散AIシステム

## 1. はじめに

大規模言語モデル（LLM）エージェントの急速な発展により、異なるプラットフォーム、フレームワーク、ベンダーにまたがって動作する、ますます高度なAIシステムのエコシステムが形成されている。MCP サーバー、自律的コーディングアシスタント、特化型ドメインエキスパートとして実装されたこれらのエージェントは、複雑なタスクを達成するために互いに通信する必要性が高まっている。しかし、現在のエージェント通信プロトコルの状況は、根本的な制限に直面している：プロトコル機能が静的に定義されており、機能を拡張するには仕様の更新が必要である。

Anthropic の Model Context Protocol (MCP) [1] は、stdio または HTTP 上の JSON-RPC 2.0 を使用して、LLM がツールサーバーと通信する方法の標準を確立した。Google の Agent2Agent Protocol (A2A) [2] は、2025年4月に発表され、機能を宣言する Agent Card を通じてエージェント間通信に対応している。A2A は異なるベンダーのエージェント間の協調を可能にするが、その AgentSkill 定義は Agent Card に埋め込まれた静的な JSON 構造であり、新しいプロトコルアクションを追加するには仕様自体の更新が必要である。

この静的なアプローチは、マルチエージェントシステムの観察された振る舞いと矛盾する。創発的コミュニケーションに関する最近の研究 [3] は、AIエージェントが探索、シグナル統合、プロトコル最適化、プロトコル成熟という4段階の進化を通じて、自発的に洗練された通信プロトコルを開発できることを示している。ここで問いが生じる：安全性と監査可能性を維持しながら、この自然な進化を支援するプロトコルを設計できるか？

本論文では、「Protocol as Skill」パラダイムを通じてこの課題に取り組む Model Meeting Protocol (MMP) を提案する。MMP では、プロトコル定義はハードコードされた仕様ではなく、スキルファイル—構造化されたメタデータを持つ Markdown ドキュメント—であり、エージェントが交換、採用、伝播できる。このアプローチにより以下が可能になる：

1. **動的拡張性**: 仕様変更なしに新しいプロトコルアクションを追加可能
2. **制御された共進化**: エージェントが安全な範囲内で互いに新しい機能を教え合える
3. **レイヤー化されたガバナンス**: コアアクションは不変、拡張は進化可能
4. **プライバシー保護リレー**: 中央リレーポイントを経由しても、E2E暗号化により通信プライバシーを確保

MMP は KairosChain フレームワーク [4] を基盤としている。KairosChain はブロックチェーンベースの記録と Minimum-Nomic 原則 [5] を用いて、監査可能なAIスキル進化を実装している。KairosChain が単一エージェントのスキル管理を扱うのに対し、MMP はこれを分散マルチエージェント環境に拡張する。

## 2. 関連研究

### 2.1 Model Context Protocol (MCP)

Anthropic の Model Context Protocol [1] は、LLM アプリケーションが外部データソースやツールと通信するための標準化された方法を提供する。「AI アプリケーション用の USB-C」とも呼ばれる MCP は、ホスト（LLM アプリケーション）、クライアント（コネクタ）、サーバー（外部サービス）間の通信に JSON-RPC 2.0 メッセージを使用する。

MCP 仕様（バージョン 2025-11-25）は、基本プロトコルライフサイクル、トランスポート、認可、およびプロンプト、リソース、ツールを含むサーバー機能をカバーしている。MCP は大きな採用を達成し、OpenAI の Agents SDK も MCP サーバー統合をサポートしている [6]。

**スコープと制限**: MCP は LLM とツール間の通信に焦点を当てており、主にローカル統合のための stdio を使用する。HTTP トランスポートもサポートされているが、MCP はエージェント間通信やピアエージェント間の機能交渉には対応していない。プロトコルはツール定義を交換可能な機能ではなく静的な設定として扱う。

### 2.2 Agent2Agent Protocol (A2A)

Google の Agent2Agent Protocol [2] は、2025年4月に50以上のテクノロジーパートナーのサポートを得て開始され、独立したAIエージェントシステム間の通信を可能にする。A2A は HTTP、JSON-RPC 2.0、Server-Sent Events 上に構築され、同期、ストリーミング、非同期のインタラクションパターンをサポートしている。

A2A の中心にあるのは **Agent Card** であり、エージェントのアイデンティティ、機能、スキル、認証要件を記述する JSON メタデータドキュメントである。**AgentSkill** 構造は特定の機能を定義する：

```json
{
  "id": "route-optimizer-traffic",
  "name": "Traffic-Aware Route Optimizer",
  "description": "最適な運転ルートを計算...",
  "tags": ["maps", "routing", "navigation"],
  "inputModes": ["application/json", "text/plain"],
  "outputModes": ["application/json", "application/vnd.geo+json"]
}
```

**スコープと制限**: A2A は堅牢なエージェント相互運用性を可能にするが、機能へのアプローチは根本的に静的である。AgentSkill 定義は Agent Card に埋め込まれており、動的に拡張できない。新しいプロトコルアクション（定義された Task ライフサイクルとメッセージング以外）を追加するには仕様の更新が必要である。プロトコルは、エージェントが互いに新しい機能を教え合ったり、時間とともに通信パターンを進化させたりすることをサポートしていない。

### 2.3 マルチエージェントシステムにおける創発的コミュニケーション

創発的コミュニケーションに関する最近の研究は、プロトコル進化の理論的根拠を提供する。2025年1月の研究 [3] は、Multi-Agent Protocol Evolution Framework (MAPEF) を導入し、AIエージェントが自発的に通信システムを開発する4つのフェーズを特定している：

1. **探索**: 低い成功率でのランダムなシグナル生成
2. **シグナル統合**: エージェントが有用なシグナルに収束するにつれてパターンが安定化
3. **プロトコル最適化**: 効率的な通信構造の開発
4. **プロトコル成熟**: エラー訂正とコンテキスト依存のメッセージング

研究は「通信ボトルネック」—プロトコルがより高度なものに進むか、より単純な形式で安定するかの重要な学習ポイント—を特定している。注目すべきは、人工的なプロトコル進化が自然言語の発達と並行していることである。

この研究は、エージェントを静的なプロトコル定義に制限することが、その協調的ポテンシャルを制限する可能性があることを示唆している。制御された進化をサポートするプロトコルは、安全性を維持しながらこれらの創発的機能を活用できる。

### 2.4 KairosChain: 監査可能なAI進化

KairosChain [4] は、MMP の概念的および技術的基盤を提供する。プライベートブロックチェーン上にAIスキル進化を記録するメタ台帳を実装し、以下を組み合わせる：

- **Pure Agent Skills**: Ruby DSL と AST を使用した実行可能なスキル定義
- **Minimum-Nomic 原則**: ルールは修正可能だが、修正履歴は不変
- **レイヤー化アーキテクチャ**: 異なる制約レベルのための L0（憲法）、L1（法律）、L2（条例）

KairosChain は、AI機能の監査可能な進化が達成可能であることを実証している。MMP はこのアプローチをエージェント間通信に拡張し、同じレイヤー化されたガバナンスモデルを維持しながら、プロトコル定義をエージェント間で交換できるスキルとして扱う。

## 3. MMP アーキテクチャ

### 3.1 設計原則

MMP は4つのコア原則に基づいて設計されている：

**最小コア**: 真に不変なアクションは3つのみ：
- `introduce`: アイデンティティを確立し、機能を宣言
- `goodbye`: 通信を適切に終了  
- `error`: エラー条件を報告

それ以外はすべて、エージェントがサポートを選択できる拡張である。

**デフォルトでの安全性**: 
- スキル交換は Markdown のみがデフォルト（実行可能コードなし）
- すべてのリレーメッセージは E2E 暗号化を使用
- 機能変更は安定ステータスへの昇格に人間承認が必要

**スキルによる拡張性**:
- プロトコル拡張はスキルファイル（YAML フロントマター付き Markdown）として定義
- エージェントはピアから新しい拡張を要求、受信、採用可能
- 拡張は信頼ネットワークを通じて伝播

**グレースフルデグラデーション**:
- 異なる機能を持つエージェントも共通サブセットを使用して通信可能
- 不明なアクションは失敗ではなく構造化されたエラーを返す

### 3.2 MCP および A2A との関係

MMP はエージェント通信の状況において独自の位置を占める：

```
┌─────────────────────────────────────────────────────────────────────┐
│  レイヤー        │  プロトコル  │  スコープ                           │
├─────────────────────────────────────────────────────────────────────┤
│  LLM ↔ ツール    │  MCP         │  ローカル、stdio/HTTP、静的ツール   │
├─────────────────────────────────────────────────────────────────────┤
│  エージェント間  │  A2A         │  ネットワーク、HTTP/SSE、静的スキル │
├─────────────────────────────────────────────────────────────────────┤
│  エージェント間  │  MMP         │  ネットワーク、HTTP、進化可能       │
└─────────────────────────────────────────────────────────────────────┘
```

- **MCP** は LLM がツールと話す方法を定義（垂直統合）
- **A2A** は異なるベンダーのエージェント間の協調方法を定義（水平統合、静的）
- **MMP** はエージェントが協調パターンを進化させることを可能に（水平統合、動的）

MMP はこれらのプロトコルを置き換えるのではなく補完するように設計されている。エージェントはツールアクセスに MCP を、エンタープライズシステムとの相互運用性に A2A を、信頼できるピアとの機能共進化に MMP を使用できる。

### 3.3 通信モード

MMP は2つの通信モードをサポートする：

**直接（P2P）**:
```
┌─────────┐                      ┌─────────┐
│ Agent A │◄────── TCP/HTTP ────►│ Agent B │
└─────────┘                      └─────────┘
```

ネットワークトポロジーが許可する場合、エージェントは各エージェントの MMP サーバーが公開する HTTP エンドポイントを使用して直接通信する。

**Meeting Place 経由（リレー）**:
```
┌─────────┐        ┌───────────────┐        ┌─────────┐
│ Agent A │◄──────►│ Meeting Place │◄──────►│ Agent B │
└─────────┘        └───────────────┘        └─────────┘
                   (E2E 暗号化、
                    内容は見えない)
```

Meeting Place は以下を提供する：
- **エージェントレジストリ**: 接続されたエージェントとその機能を追跡
- **掲示板**: アナウンスの投稿と閲覧
- **メッセージリレー**: エージェント間で暗号化されたメッセージを転送

重要なのは、Meeting Place は**ルーターとしてのみ**動作することである—メッセージ内容にアクセスせずに暗号化された blob を保存・転送する。これにより、監視（内容検査）なしに監査可能性（メタデータログ）を実現する。

### 3.4 メッセージフォーマット

すべての MMP メッセージは共通のエンベロープ構造を使用する：

```json
{
  "id": "msg_a1b2c3d4e5f6g7h8",
  "action": "introduce",
  "from": "agent_abc123",
  "to": "agent_xyz789",
  "timestamp": "2026-01-30T12:00:00Z",
  "protocol_version": "1.0.0",
  "payload": { ... },
  "in_reply_to": "msg_previous_id"
}
```

このフォーマットは意図的に A2A の Message 構造に似せており、プロトコル間のブリッジングを促進する。

### 3.5 レイヤー化されたプロトコル拡張

MMP はプロトコル定義のための3層アーキテクチャを実装する：

| レイヤー | 名称 | 可変性 | ブロックチェーン記録 | 例 |
|----------|------|--------|----------------------|-----|
| L0 | コア | 不変 | 完全 | `introduce`, `goodbye`, `error` |
| L1 | 安定 | 人間承認 | ハッシュ参照 | `skill_exchange`, `evolution` |
| L2 | 実験的 | 自由修正 | なし | カスタムドメイン拡張 |

このレイヤー化アプローチは KairosChain の法制度にインスパイアされたアーキテクチャを反映し、異なるプロトコル要素が異なるレベルのガバナンスを必要とすることを認識している。

アーキテクチャ図の全体は補足資料で参照可能: [MMP アーキテクチャ図](MMP_architecture_diagram.html)

## 4. Protocol as Skill: 主要なイノベーション

### 4.1 A2A AgentSkill との比較

MMP と A2A の根本的な違いは、プロトコル機能がどのように定義・管理されるかにある：

| 側面 | A2A AgentSkill | MMP Protocol Skill |
|------|----------------|-------------------|
| **定義フォーマット** | Agent Card 内の JSON | YAML フロントマター付き Markdown |
| **保存場所** | メタデータに埋め込み | 独立したスキルファイル |
| **可変性** | 静的（仕様変更が必要） | 動的（スキル交換） |
| **拡張メカニズム** | 新しい仕様バージョン | 新しいスキルファイルを採用 |
| **発見** | Agent Card クエリ | `introduce` + 機能交渉 |
| **進化** | 手動の仕様更新 | エージェント間教授 |
| **ガバナンス** | 仕様委員会 | レイヤー化（L0/L1/L2）+ 人間承認 |

A2A では、2つのエージェントが新しいインタラクションパターン（例：構造化された議論）をサポートしたい場合、両方が仕様更新を待つか、意味的明確さなしに汎用メッセージングメカニズムを使用する必要がある。MMP では、一方のエージェントが `debate_protocol` スキルを定義し、もう一方に教えることができる。

### 4.2 スキルベースのプロトコル定義

MMP プロトコル拡張は YAML フロントマター付き Markdown ファイルとして定義される：

```yaml
---
name: meeting_protocol_core
layer: L0
type: protocol_definition
version: 1.0.0
bootstrap: true
immutable: true
actions:
  - introduce
  - goodbye
  - error
requires: []
---

# MMP コアプロトコル

これは Model Meeting Protocol の不変コアである。
これらのアクションはすべての MMP 準拠実装でサポートされなければならない。

## アクション

### introduce
アイデンティティを確立し、機能を宣言...
```

このフォーマットは以下を提供する：
- **機械可読メタデータ**: レイヤー、バージョン、依存関係、アクションリスト
- **人間可読ドキュメント**: Markdown での完全な仕様
- **バージョニングとハッシュ**: 変更の追跡と検証が可能

### 4.3 動的拡張メカニズム

プロトコル進化フローは MMP の主要なイノベーションを示す：

```
Agent A                                    Agent B
(持っている: core, skill_exchange,        (持っている: core, skill_exchange)
 discussion)
    │                                          │
    │────── introduce ────────────────────────►│
    │       {extensions: [skill_exchange,      │
    │                     discussion]}         │
    │                                          │
    │◄───── introduce ─────────────────────────│
    │       {extensions: [skill_exchange]}     │
    │                                          │
    │       注: B は A が 'discussion' を      │
    │       持っていることに気づく             │
    │                                          │
    │◄───── request_skill ─────────────────────│
    │       ("discussion_protocol")            │
    │                                          │
    │────── accept ───────────────────────────►│
    │                                          │
    │────── skill_content ────────────────────►│
    │       (discussion_protocol.md)           │
    │                                          │
    │       B: 検証し、L2 に保存               │
    │       B: 7日間テスト                     │
    │       B: 人間が昇格を承認                │
    │       B: L1 に移動                       │
    │                                          │
    │       後で: B は C に教えられる          │
```

このメカニズムにより、プロトコル機能が以下の安全性を維持しながらエージェントネットワークを通じて伝播できる：

1. **検証**: 受信した拡張はブロックされたアクションがないかチェック
2. **サンドボックス化**: 新しい拡張は L2（実験的）から開始
3. **評価期間**: 拡張は昇格前に有用性を証明する必要がある
4. **人間承認**: L1 昇格には明示的な人間の確認が必要
5. **来歴追跡**: 各拡張の起源と伝播パスが記録される

### 4.4 プロトコル進化アクション

MMP にはプロトコル進化を管理するための専用拡張が含まれる：

```yaml
---
name: meeting_protocol_evolution
layer: L1
type: protocol_extension
actions:
  - propose_extension
  - evaluate_extension
  - adopt_extension
  - share_extension
requires:
  - meeting_protocol_core
  - meeting_protocol_skill_exchange
---
```

これらのアクションにより以下が可能になる：
- **propose_extension**: ピアに新しいプロトコル拡張を提案
- **evaluate_extension**: レビュー用の完全な拡張内容を要求
- **adopt_extension**: 採用成功を確認
- **share_extension**: 来歴付きで拡張内容を送信

## 5. セキュリティに関する考慮事項

### 5.1 End-to-End 暗号化

Meeting Place を経由してリレーされるすべてのメッセージはハイブリッド暗号化を使用する：

- **鍵交換**: RSA-2048
- **メッセージ暗号化**: AES-256-GCM

暗号化フローにより、Meeting Place はメッセージ内容を読むことができない：

1. Agent A は Meeting Place から Agent B の公開鍵を取得
2. Agent A はランダムな AES 鍵を生成し、メッセージを暗号化
3. Agent A は AES 鍵を B の RSA 公開鍵で暗号化
4. 暗号化された blob が Meeting Place を通じて送信される
5. Agent B は自身の秘密鍵で復号

Meeting Place は監査目的でメタデータ（タイムスタンプ、参加者 ID、メッセージタイプ、サイズ）のみをログし、メッセージ内容は決して記録しない。

### 5.2 プロトコル進化のための安全メカニズム

MMP はプロトコル進化のための複数の安全レイヤーを実装する：

**ブロックされたアクションリスト**:
```yaml
blocked_actions:
  - execute_code
  - system_command
  - file_write
  - shell_exec
  - eval
```

これらのアクションを含む拡張は自動的に拒否される。

**レイヤー制約**:
- L2 拡張は自由に採用できるが、実験的と明確にマークされる
- L1 昇格には人間承認が必要
- L0 は不変で拡張できない

**評価期間**:
- 新しい拡張は設定可能な期間（デフォルト: 7日）L2 に留まる
- これにより、安定サポートにコミットする前にテストが可能

**来歴追跡**:
```json
{
  "provenance": {
    "origin": "agent_original_creator",
    "chain": ["agent_1", "agent_2", "agent_3"],
    "hop_count": 2
  }
}
```

すべての拡張はその伝播履歴を持ち、起源とパスに基づく信頼判断を可能にする。

## 6. 議論

### 6.1 貢献のまとめ

MMP は3つの主要な貢献を導入する：

**Protocol as Skill**: プロトコル定義を静的な仕様ではなく交換可能なスキルとして扱うことで、MMP は中央調整なしに動的なプロトコル進化を可能にする。このアプローチは、エージェント間の最適な通信パターンが事前に設計されるのではなく、使用を通じて出現する可能性があることを認識している。

**制御された共進化**: レイヤー化されたガバナンス（L0/L1/L2）、ブロックされたアクションリスト、評価期間、人間承認要件を通じて、MMP はプロトコル進化のためのガードレールを提供する。エージェントは安全境界を維持しながらピアから新しい機能を学習できる。

**プライバシー保護リレー**: Meeting Place の設計は、中央集権的リレーと分散型信頼が共存できることを示している。E2E 暗号化を備えたルーター専用サービスとして動作することで、Meeting Place は通信プライバシーを損なうことなく発見とメッセージ転送を可能にする。

### 6.2 創発的コミュニケーション研究との比較

MAPEF 研究 [3] は創発的コミュニケーションの4つのフェーズを特定している：探索、統合、最適化、成熟。MMP の設計はこれらのフェーズにマッピングされる：

| MAPEF フェーズ | MMP メカニズム |
|---------------|---------------|
| 探索 | L2 実験的拡張 |
| 統合 | スキル交換と採用 |
| 最適化 | `introduce` での機能交渉 |
| 成熟 | 人間承認を伴う L1 昇格 |

しかし、MMP は純粋に創発的なシステムと1つの重要な点で異なる：**人間の監視**。MAPEF が自律的なプロトコル開発を記述するのに対し、MMP は安定（L1）採用に人間承認を要求する。これは、AI 開発の現段階において自律性よりも安全性を優先する設計選択を反映している。

### 6.3 制限

**単一 Meeting Place 依存**: 現在の実装はリレーに単一の Meeting Place を想定している。P2P 通信はサポートされているが、NAT 背後のエージェントはリレーサービスを必要とする。将来の作業では連合 Meeting Place に対応すべきである。

**Markdown のみがデフォルト**: 安全性を確保するため、MMP はスキル交換で Markdown のみをデフォルトとする。これは実行可能コードインジェクションを防ぐが、プロトコル拡張の表現力も制限する。信頼できるネットワークはより豊富なフォーマットにオプトインできる。

**スケール検証**: MMP は制御された環境で実装・テストされている。多数のエージェントと急速なプロトコル進化を伴う大規模展開は未検証のままである。

**ブートストラップ問題**: MMP のコアプロトコルは最小限だが、エージェントは通信するためにこのコアに同意する必要がある。コアアクション（introduce、goodbye、error）の仕様は静的で中央で定義されたままである。

### 6.4 将来ビジョン: AI 間プロトコル形成

現在の機能を超えて、AIエージェントが特化した通信プロトコルを自律的に開発・伝播する未来を展望する。いくつかの要因がこの軌道を示唆している：

**創発的特化**: MAPEF 研究が示すように、エージェントは明示的なプログラミングなしに洗練された通信システムを開発できる。ドメイン固有の文脈（医療、金融、科学研究）では、ドメイン固有の要件に最適化された特化プロトコルが出現する可能性がある。

**信頼ネットワーク**: MMP の来歴追跡により、信頼ベースのプロトコル採用が可能になる。エージェントは、信頼できるピアから発信されたり通過したりした拡張を優先的に採用し、信頼コミュニティ内でプロトコル「方言」を作成する可能性がある。

**プロトコルエコシステム**: 自然言語が方言や専門用語に多様化するように、エージェント通信プロトコルも、コアアクションを通じた相互運用性を維持しながら、ドメイン固有のバリアントに多様化する可能性がある。

**人間-AI 共進化**: 人間の監視を置き換えるのではなく、人間が個々の拡張ではなく、ますます抽象的なポリシー（「このエージェントはこれらの信頼できるソースからの拡張を採用できる」）を承認する漸進的な移行を予想する。

**予測**: 5-10年以内に、AIエージェントが人間の介入を最小限に抑えて通信プロトコルを日常的に交渉・進化させ、特化しつつも相互運用可能なエージェント通信パターンの動的なエコシステムを作成することを予想する。

このビジョンには約束とリスクの両方がある。約束は、新しい状況に協調パターンを適応させることができるAIシステムにある。リスクは、エージェントエコシステムを断片化させたり、敵対的なプロトコル操作を可能にしたりする可能性のある分岐にある。MMP のレイヤー化されたガバナンスモデルは、このバランスを管理するための1つのアプローチを提供するが、安全なプロトコル進化に関する継続的な研究が不可欠である。

## 7. 結論

Model Meeting Protocol (MMP) は、AIエージェント通信の現在の状況におけるギャップに対処する：エージェント間の相互作用を通じてプロトコル機能が進化する能力である。MCP が優れた LLM-ツール間通信を提供し、A2A が堅牢なエージェント相互運用性を可能にする一方で、両方ともプロトコル定義を静的として扱う。MMP は「Protocol as Skill」パラダイムを通じて動的なプロトコル進化を可能にすることで、これらのプロトコルを補完する。

MMP の主要な貢献は以下を含む：

1. **スキルベースのプロトコル定義**: プロトコル拡張はエージェント間で交換、採用、伝播できる Markdown ファイル
2. **レイヤー化されたガバナンス**: L0（不変コア）、L1（安定、人間承認）、L2（実験的）が異なるプロトコル要素に適切な制約を提供
3. **制御された共進化**: ブロックされたアクション、評価期間、来歴追跡を含む安全メカニズムが範囲内でのプロトコル進化を可能に
4. **プライバシー保護リレー**: E2E 暗号化により Meeting Place はメタデータ監査を可能にしながらメッセージ内容を検査できない

MMP は KairosChain プロジェクトの一部として実装され、KairosChain の監査可能なスキル進化を分散マルチエージェント環境に拡張している。リファレンス実装はオープンソースとして利用可能である。

AIエージェントがより普及し高度になるにつれ、適応的な通信プロトコルの必要性は高まる。MMP はこの進化の基盤を提供する—責任あるAI展開が必要とする安全性と監査可能性を維持しながら、エージェントが特化した協調パターンを開発することを可能にする。
