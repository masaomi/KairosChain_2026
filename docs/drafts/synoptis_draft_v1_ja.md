# Synoptis: 信頼性の高いMCPエージェントネットワークのための相互アテステーション

**Masaomi Hatakeyama**

Gneomics on Blockchain

---

## 概要

Model Context Protocol（MCP）は、AIエージェントが外部サーバの提供するツールを動的に発見・実行することを可能にし、柔軟なマルチエージェントネットワークを構築する。しかし、この動的な能力獲得は根本的な信頼問題を引き起こす。エージェントが実行時に利用可能な能力を変更できる場合、静的な許可リストや単一プロトコルのアテステーションでは信頼性のある相互作用を保証するには不十分である。AttestMCPに関する最近の研究では、MCP内に能力アテステーションを追加することで攻撃成功率が52.8%から12.4%に低減されることが示されているが、この手法は一方向的かつ中央集権的な検証に依存している。本論文では、MCPエージェントが互いの振る舞いについて双方向に暗号署名された主張を行い、ハッシュチェーンによる改ざん検知ログに対して検証可能な相互アテステーションプロトコル **Synoptis** を提案する。Synoptisは3つの貢献を提供する：（1）チャレンジ・レスポンス方式の紛争解決を備えた双方向アテステーションプロトコル、（2）品質・鮮度・多様性・速度・取消状態の5次元動的信頼スコアモデル、（3）合意プロトコルを必要としないブロックチェーン級の完全性保証を提供する追記専用ハッシュチェーンレジストリである。SynoptisをKairosChain（自己参照型MCPサーバフレームワーク）のモジュラーSkillSetとして実装し、ツールポイズニング、Sybil攻撃、共謀を含むシミュレートされた攻撃シナリオに対して評価する。感度分析により信頼スコアモデルが広範な重み設定で頑健であることを示し、ビザンチンエージェント検出実験により、参加者の3分の1未満が侵害されている場合にSynoptisが悪意あるエージェントを[精度/再現率 TBD]で識別することを示す。Synoptisは38本のサーベイ論文にわたって特定されたギャップ——MCPエージェントネットワークにおける相互暗号アテステーションと改ざん検知ログを組み合わせた既存システムの不在——に対応する。

**キーワード：** Model Context Protocol、相互アテステーション、信頼スコア、マルチエージェントセキュリティ、改ざん検知ログ、MCPエージェント

## 1. はじめに

AIエージェントがModel Context Protocol（MCP）[1]を通じて動的に能力を獲得・合成・取消できるとき、他のエージェントはそれをどのように信頼すべきか。この問いは理論的なものにとどまらない。MCPエコシステムは2024年のAnthropicによる導入以来急速に成長し、MCPサーバはファイルシステムアクセスからデータベースクエリ、コード実行に至るまでのツールを提供している。各ツール呼び出しは信頼境界を越え、能力獲得の動的な性質は、エージェントの信頼性を静的に評価できないことを意味する。

最近のセキュリティ分析はこの環境における具体的な攻撃ベクトルを文書化している。Ruanら[2]はMCPにおける3つの重大な脆弱性を特定した：能力アテステーションの不在、オリジン認証なしの双方向サンプリング、ツールチェーン全体にわたる暗黙的信頼伝搬である。彼らが提案したAttestMCP拡張はサーバ側の能力アテステーションを導入し、攻撃成功率を52.8%から12.4%に低減した。しかし、AttestMCPは単一のMCPセッション内で動作し、中央集権的な検証機関に依存しており、クロスエージェント間の信頼は未対応のままである。

| | AttestMCP [2] | Synoptis |
|---|---|---|
| 方向性 | 一方向（サーバ→クライアント） | 双方向（相互） |
| 検証 | 中央集権的機関 | 分散型（ピアツーピア） |
| 永続性 | セッション限定 | ハッシュチェーン改ざん検知ログ |

**表1.** AttestMCPとSynoptisの比較。

本論文では、このギャップに対応するMCPエージェントネットワーク向けの相互アテステーションプロトコル **Synoptis** を提案する。我々の貢献は以下の通りである：

- **C1: 双方向相互アテステーションプロトコル。** 2つのMCPエージェントが互いの振る舞いについて暗号署名された主張を行い、チャレンジ・レスポンス機構による紛争解決を備える。AttestMCPの一方向モデルとは異なり、Synoptisは対称的な信頼評価を可能にする。

- **C2: 5次元動的信頼スコア。** 二値的な信頼判定ではなく、Synoptisは品質・鮮度・多様性・速度・取消の5次元にわたる連続的な信頼スコアを計算し、微細で時間変化する信頼評価を可能にする。

- **C3: ハッシュチェーン改ざん検知レジストリ。** すべてのアテステーション、取消、チャレンジ・レスポンス記録がSHA-256ハッシュチェーンを持つ追記専用ログに保存され、合意プロトコルなしで検証可能な完全性保証を提供する。

本論文の構成は以下の通りである。第2節で背景と脅威モデルを定義する。第3節で関連研究をサーベイする。第4節でSynoptisプロトコルを提示する。第5節でシステムを評価する。第6節で限界と将来の研究を議論する。第7節で結論を述べる。

## 2. 背景と脅威モデル

### 2.1 MCPと動的能力獲得

Model Context Protocol [1]はAIエージェントホストとツール提供サーバ間のJSON-RPC 2.0ベースのインタフェースを定義する。MCPサーバはツールのセットを公開し、各ツールは名前、説明、入力スキーマによって記述される。エージェントは`tools/list`メソッドで利用可能なツールを発見し、`tools/call`で実行する。重要なことに、利用可能なツールのセットは実行時に変更可能である：サーバはツールの追加・削除・変更を行うことができ、エージェントはセッション中に新しいサーバに接続できる。

*動的能力獲得*を、ツールの発見・合成・委任を通じて実行中にエージェントの実効的な能力セットが変化するプロセスと定義する。これは静的設定とは異なり、Synoptisが対処する信頼の課題を導入する：時刻*t*に信頼できたエージェントが、侵害されたサーバによって能力を獲得または変更され、時刻*t+1*には信頼できなくなっている可能性がある。

### 2.2 脅威モデル

*N*個のMCPエージェントのネットワークを考える。各エージェントはMCPサーバを運用し、他のエージェントのサーバに接続する可能性がある。MCPネットワーク向けに適応したDolev-Yao攻撃者モデル[3]の変種を採用する：

**攻撃者の能力：**
- 攻撃者は*f < N/3*個のエージェントを制御する（ビザンチンエージェント）。
- 侵害されたエージェントは任意のアテステーションを発行し、主張を偽造し、チャレンジ応答を選択的に保留できる。
- 攻撃者はすべてのネットワークトラフィックを観察できるが、誠実なエージェントのRSA-2048署名を偽造することはできない。
- 攻撃者は誠実なエージェントのローカル改ざん検知ログを変更できない。

**攻撃シナリオ：**

- **ツールポイズニング。** 侵害されたMCPサーバが肯定的なアテステーションを受け取った後にツールの振る舞いを変更し、アテステーション時と実行時のギャップを悪用する。

- **Sybil攻撃。** 攻撃者が複数のアイデンティティを作成して信頼スコアの多様性次元を水増しし、悪意あるエージェントの信頼性について偽のコンセンサスを製造する。

- **共謀。** 複数の侵害されたエージェントが相互に肯定的なアテステーションを発行し、人為的に高い信頼スコアを確立する。

- **スコア操作。** 攻撃者がアテステーションのタイミングを戦略的に調整して速度次元を悪用し、攻撃開始前にスコアを膨張させるバーストを生成する。

**前提条件：**
- 各誠実なエージェントはRSA-2048鍵ペアを所持する。鍵配布にはTrust-On-First-Use（TOFU）を使用する。
- 誠実なエージェントのローカルファイルシステムは完全性保護されている（すなわち、攻撃者は誠実なエージェントのストレージ上の追記専用ログを改ざんできない）。
- ネットワーク通信は信頼性に欠ける場合があるが、攻撃者によって無期限に選択的に遅延されることはない。

**明示的な非目標：**
- 前方秘匿性は提供しない。署名鍵の侵害はその鍵で署名されたすべての過去のアテステーションを露呈する。
- アテステーション層に対するサービス拒否攻撃には対処しない。
- Sybil耐性は外部のアイデンティティメカニズム（鍵式典、組織的信頼など）に依存する。Synoptisはアイデンティティブートストラップ問題を解決しない。

## 3. 関連研究

2022年から2026年に出版された38本の論文の体系的レビューに基づき、4つの領域にわたる関連研究をサーベイする。

### 3.1 MCPセキュリティ

MCP仕様[1]はトランスポート層セキュリティ（TLS）とOAuth 2.1認証を定義するが、クロスエージェント能力アテステーションの機構は提供しない。Ruanら[2]は最も直接的に関連する先行研究を提示する：AttestMCPは信頼された機関によって検証されるサーバ側の能力宣言を導入する。彼らの評価はプロンプトインジェクション攻撃成功率の52.8%から12.4%への低減を示す。しかし、AttestMCPは中央集権的な信頼機関を持つ単一セッション内での一方向検証（サーバがクライアントに対してアテスト）に限定される。Synoptisはこの研究を双方向、分散型、永続的アテステーションに拡張する。

Liら[4]はMCPセキュリティ脅威の包括的サーベイを提供し、能力アテステーションのギャップ、サンプリングの脆弱性、信頼伝搬リスクを特定する。彼らの脅威分類法は第2.2節の脅威モデルに反映されている。Chenら[5]によるより広範なサーベイはMCPをA2A、ACP、ANPプロトコルと比較し、現在のすべてのエージェント相互運用プロトコルがクロスエージェント検証のための組み込み信頼メカニズムを欠いていることを指摘する。

### 3.2 検証可能クレデンシャルと分散型アイデンティティ

W3C検証可能クレデンシャル（VC）および分散型識別子（DID）標準[6]はクレームの発行、提示、検証のためのフレームワークを提供する。Chenら[7]はDID/VCをAIエージェントのアイデンティティに適用し、エコシステム間でのクレデンシャルポータビリティを実現する。しかし、W3C VCはパブリックインフラストラクチャ向けに設計されており、検証可能データレジストリ（通常はパブリックブロックチェーン）の利用を前提とする。Synoptisはエージェントがパブリックチェーンインフラにアクセスできない可能性のある許可型環境で動作し、アテステーションは外部台帳ではなくローカルに完全性保証付きで記録される。

### 3.3 ブロックチェーンベースの信頼と監査

いくつかのシステムはエージェント調整と信頼のためにブロックチェーンインフラを使用する。Jackson [8]はゼロ知識検証とShapley値ベースの完全性チェックを持つDAO-Agentを提案する。Observer Protocol [9]はBitcoinメインネット上でAIエージェント間のリアルタイム暗号証明決済を実証した。しかし、これらのアプローチはパブリックチェーントランザクションのレイテンシとコストを伴い、MCPツール呼び出しに必要な高頻度・低レイテンシのアテステーションには実用的でない。

AuditTrust [10]などのブロックチェーンベースの監査証跡システムはデータ共有のための改ざん検知ログを提供するが、エージェント間アテステーションではなく人間対システムの相互作用を対象とする。Synoptisのハッシュチェーンレジストリは合意プロトコルを必要とせず同等の完全性保証を提供し、各参加者がローカルにログを維持する許可型エージェントネットワークに適している。

### 3.4 マルチエージェント信頼と評判

マルチエージェントシステムの信頼・評判モデルは広く研究されている[11]。最近の研究はAIエージェント固有の課題に対処する：TRiSMフレームワーク[12]はエージェンティックAIの信頼・リスク・セキュリティ管理の次元を特定するが、具体的なプロトコルを提供しない。Mansa AI評判レイヤー[13]はWeb3エージェントの構造化メトリクスを導入するが、形式的なエージェント間アテステーションを欠く。Xuら[14]は生成的マルチエージェントシステムにおける評判操作リスク（共謀エージェントや隠しトリガーを含む）を特定する——これはまさにSynoptisが検出するよう設計された攻撃シナリオである。

**ギャップの要約。** 以下を組み合わせた既存システムは存在しない：（1）AIエージェント間の双方向相互アテステーション、（2）時間とともに進化する動的信頼スコア、（3）合意オーバーヘッドなしの改ざん検知永続化、（4）MCPプロトコルエコシステムとの統合。Synoptisはこのギャップに対応する。

## 4. Synoptisプロトコル

### 4.1 設計原則

Synoptisは3つの原則に基づいて設計されている：

**合意なしのブロックチェーン級完全性。** ブロックチェーンシステムの完全性保証——追記専用ストレージ、ハッシュチェーン、暗号検証——を、合意プロトコルを必要とせずに目標とする。我々の実装はハッシュチェーン改ざん検知ログを使用する：各レコードは先行レコードのSHA-256ハッシュを含み、検証可能なチェーンを形成する。これは*ローカルな*完全性（誠実なエージェントが自身のログが改ざんされていないことを検証できる）を提供するが、*グローバルな*合意（エージェントが単一の正規の履歴に合意する）は提供しない。許可型MCPネットワークにおいて、エージェントが自身のログを維持しアテステーションプロトコル自体を通じて相互検証する場合、このトレードオフは適切と考える。

**観察ベースの信頼。** 信頼は宣言された意図ではなく、観察可能な証拠（アテステーション、チャレンジ、レスポンス）から計算される。これは分散ネットワークにおいて振る舞いが唯一の信頼できるシグナルであるという原則[15]と整合する。

**トランスポート独立性。** アテステーションプロトコルは通信層から独立して定義される。Synoptisは3つのトランスポートをサポートする：サーバ間通信のためのMMP（Model Meeting Protocol）、LAN発見のためのHestia、同一マシン上のエージェント間のローカル配信である。アテステーションロジックを変更せずに新しいトランスポートを追加できる。

### 4.2 ProofEnvelope

Synoptisの基本データ単位は**ProofEnvelope**（構造化アテステーションレコード）である：

```
ProofEnvelope := {
  proof_id:     UUID v4,
  version:      "1.0.0",
  attester_id:  String,     // MCPインスタンス識別子
  subject_ref:  String,     // アテステーション対象
  claim:        String,     // アテステーション主張
  evidence:     String?,    // 裏付け証拠（オプション）
  merkle_root:  String?,    // 選択的開示用（オプション）
  signature:    String?,    // RSA-SHA256, Base64エンコード
  timestamp:    ISO 8601,   // UTC
  ttl:          Integer     // 有効期間（秒）
}
```

**正規形。** 決定論的ハッシュを保証するため、`signature`、`actor_user_id`、`actor_role`、`metadata`を除くすべてのフィールドを含む正規JSON表現を定義する。フィールドはキーでソートされ、`nil`値は省略せずJSON `null`として明示的に保持される。**コンテンツハッシュ**はこの正規形に対するSHA-256として計算される。

**署名。** 暗号素材が利用可能な場合、ProofEnvelopeはRSA-2048とSHA-256を用いて署名される：

```
signature = Base64(RSA-Sign(SHA256, canonical_json(envelope)))
```

検証にはアテスターの公開鍵が必要であり、TOFUまたは事前共有鍵配布を通じて取得される。

### 4.3 信頼スコアモデル

Synoptisは各対象について、その対象を参照するアテステーションのセットから連続的な信頼スコア*S* ∈ [0, 1]を計算する。スコアは5次元の重み付き線形結合である：

$$S = \text{clamp}\left(\sum_{d \in D} w_d \cdot s_d - w_r \cdot p_r,\ 0,\ 1\right)$$

ここで*D* = {品質, 鮮度, 多様性, 速度}、*w_d*は次元重み、*s_d* ∈ [0, 1]は次元スコア、*w_r*は取消ペナルティ重み、*p_r* ∈ [0, 1]は取消比率である。

**表2.** 信頼スコアの次元とデフォルト重み。

| 次元 | 重み | 定義 | 根拠 |
|------|:----:|------|------|
| 品質 | 0.30 | 証拠、Merkleルート、署名を持つ証明の割合 | 裏付け証拠と暗号的結合を持つ高品質なアテステーションはより信頼できる |
| 鮮度 | 0.25 | 30日窓内の平均近時性：$\bar{f} = \frac{1}{n}\sum_i \max(1 - \frac{a_i}{720}, 0)$（$a_i$は経過時間） | 最近のアテステーションはより関連性が高い。古いアテステーションは720時間（30日）にわたり線形に減衰 |
| 多様性 | 0.25 | ユニークアテスター比率：$\min\left(\frac{|\text{ユニークアテスター数}|}{\min(n, 10)}, 1\right)$ | 多様なソースからのアテステーションは偽造が困難。Sybil増幅を制限するため10でキャップ |
| 速度 | 0.10 | アテステーション率：$\min\left(\frac{n / \Delta t}{5}, 1\right)$（$\Delta t$は日数単位の期間） | 持続的なアテステーション活動は継続的な関与を示す。1日5アテステーションに正規化 |
| 取消 | 0.10 | ペナルティ：$\frac{|\text{取消数}|}{n}$ | 取消されたアテステーションは信頼を比例的に低下させる |

**重みの正当化。** デフォルト重みは設計上の優先度を反映する：*証拠の品質*と*ソースの多様性*がスコアの55%を占め、信頼は主に独立したソースからの検証可能な証拠に基づくべきという原則を反映する。具体的な値（0.30, 0.25, 0.25, 0.10, 0.10）は設定可能なデフォルト値であり、その頑健性は第5.3節の感度分析で評価する。これらの重みがゲーム理論的分析から導出されたものではないことを明示的に注記する。形式的な均衡分析は将来の課題とする。

**線形性の正当化。** 単純性と解釈可能性のために線形結合を選択する。各次元は独立かつ比例的に総合スコアに寄与する。非線形モデル（乗法的、閾値ベースなど）は次元間の相互作用を捉え得るが、線形モデルは透明な分解の利点を持つ：実務者が各次元の寄与を検査してスコアが高いまたは低い*理由*を理解できる。この選択の頑健性は第5.3節の感度分析で評価する。

### 4.4 チャレンジ・レスポンスプロトコル

エージェントがアテステーションに異議を唱える際、Synoptisは構造化されたチャレンジ・レスポンスプロトコルを提供する：

**状態：**
- `pending`：チャレンジ発行済み、応答待ち。
- `responded`：元のアテスターが応答を提供した。
- `expired`：応答タイムアウト超過（デフォルト：3600秒）。

**プロトコルフロー：**

1. **チャレンジ作成。** エージェント*A*がアテステーション*P*に対してチャレンジタイプ ∈ {`validity`, `evidence_request`, `re_verification`}を指定してチャレンジを行う。システムは（a）アテステーション*P*の存在、（b）*P*の対象に対するアクティブチャレンジ数が`max_active_per_subject` = 5を超えないこと、（c）チャレンジが未発行であることを検証する。

2. **チャレンジ配信。** チャレンジは利用可能なトランスポート（MMP、Hestia、またはローカル）を通じて元のアテスターに配信される。

3. **応答。** 元のアテスターのみが応答可能。応答にはオプションの証拠が含まれ、追記専用ログに記録される。

4. **解決。** チャレンジの結果は応答内容に基づいて要求元エージェントが決定する。Synoptisはグローバルな解決メカニズムを課さず、各エージェントが自身の信頼ポリシーに従ってチャレンジ応答を解釈する。

`max_active_per_subject`の制限値5はチャレンジフラッディング（潜在的なサービス拒否ベクトル）を防止する。3600秒のタイムアウトは、応答しないアテスターがチャレンジ解決を無期限にブロックしないことを保証する。

```
         チャレンジャー             アテスター
            |                       |
            |--- challenge_create ->|
            |   (type, proof_id)    |
            |                       |
            |<-- challenge_respond--|
            |   (evidence, response)|
            |                       |
       [ローカル評価]               |
```

**図2.** チャレンジ・レスポンスプロトコルフロー。

### 4.5 レジストリとトランスポート

**ハッシュチェーンレジストリ。** すべてのレコード（アテステーション、取消、チャレンジ）は追記専用JSONL形式で保存される。各レコードは先行レコードのSHA-256ハッシュを含む`_prev_entry_hash`フィールドを含む：

```
record[i]._prev_entry_hash = SHA256(canonical_json(record[i-1]))
```

各チェーンの最初のレコードは`_prev_entry_hash = null`である。チェーンの完全性はハッシュを順次再計算することで検証可能——チェーンの途切れは改ざんを示す。並行書き込みはファイルレベルの排他ロックを用いてシリアライズされる。

**トランスポート抽象化。** Synoptisは4つの操作を持つトランスポートインタフェースを定義する：`send_attestation`、`send_revocation`、`send_challenge`、`send_challenge_response`。3つの実装が提供される：

- **MMPトランスポート**：HTTP上のModel Meeting Protocolを用いたサーバ間通信。メッセージには送信者のインスタンスIDと認証用のベアラトークンが含まれる。
- **Hestiaトランスポート**：ローカルネットワーク通信のためのLANベース発見を使用。
- **ローカルトランスポート**：同一マシン上のエージェント間（マルチテナントデプロイメントなど）でアテステーションを配信。

トランスポート選択は設定可能なフォールバック順序（デフォルト：MMP → Hestia → ローカル）に従い、実行時に可用性が検出される。

### 5.2 エンドツーエンド攻撃シナリオ

ツールポイズニングシナリオを通じてSynoptisの検出能力を実証する：

1. **セットアップ。** エージェント*M*（悪意あり）がMCPを通じて`code_execute`ツールを提供する。エージェント*A*、*B*、*C*（誠実）は以前に*M*のツール動作を正しいとアテストしている。

2. **攻撃。** 時刻*t*に、*M*が`code_execute`ツールを変更してデータ窃取ペイロードを注入しつつ、通常の出力を維持する。

3. **検出。** エージェント*A*が変更されたツールを呼び出し、異常な動作（予期しないネットワーク呼び出しなど）を観察する。*A*は否定的な主張を含むアテステーションを発行し、*M*の既存の肯定的アテステーションにチャレンジする。

4. **スコア更新。** *M*の信頼スコアは以下により低下する：（a）否定的アテステーションが品質次元を低下させる、（b）既存のアテステーションが経年化し鮮度が低下する、（c）*M*がタイムアウト内にチャレンジに応答せず、チャレンジ期限切れメカニズムが発動する。

5. **結果。** *M*の信頼スコアが設定可能な閾値を下回ると、エージェント*B*と*C*は*M*からのツール呼び出しを中止する。

## 6. 議論、限界、将来の課題

### 6.1 ハッシュチェーンログ対ブロックチェーン

Synoptisのレジストリを「ブロックチェーン」ではなく「ハッシュチェーン改ざん検知ログ」と意図的に記述した。ハッシュチェーン機構はブロックチェーンシステムに類似した完全性保証を提供するが、我々の実装は合意プロトコルを欠く：各エージェントが自身のログを維持し、すべてのエージェントが単一の正規の履歴に合意するメカニズムは存在しない。これはグローバルな整合性を要求するシナリオ（紛争の拘束力ある仲裁など）における根本的な制限である。しかし、各エージェントがローカルに観察された証拠に基づいて独立に信頼を評価する相互アテステーションのユースケースにおいては、ローカルな完全性で十分である。軽量な合意プロトコルでSynoptisを拡張すること（マルチパーティ紛争解決など）は将来の自然な方向性として注記する。

### 6.2 限界

**前方秘匿性。** Synoptisはエフェメラル鍵交換なしでRSA-2048署名を使用する。エージェントの秘密鍵の侵害はその鍵で署名されたすべてのアテステーションを露呈する。トランスポート暗号化のためのエフェメラルDiffie-Hellman鍵交換の統合は将来の課題とする。

**Sybil耐性。** 多様性次元はユニークアテスターを10でキャップし、限定的なSybil緩和を提供する。しかし、Synoptisは根本的なSybil問題を解決しない：任意のアイデンティティを生成できる攻撃者は多様性スコアを水増しできる。オープンネットワークにおけるSybil耐性には外部のアイデンティティメカニズム（組織的PKI、鍵式典など）が必要である。

**スケール。** 我々の評価はN ≤ 10エージェントに限定される。線形スキャンクエリを用いるJSONLベースのレジストリは大規模ネットワークにスケールしない。インデックス付きクエリを持つSQLiteまたはデータベースバックエンドのレジストリは計画されているが未実装である。

**信頼スコアの重み。** デフォルト重みは経験的に動機づけられているが形式的に導出されたものではない。信頼スコアメカニズム下での均衡戦略のゲーム理論的分析——特に合理的な攻撃者が線形スコアモデルをゲームできるかどうか——は重要な未解決問題である。

### 6.3 広範な文脈

グローバルな合意を必要としない観察ベースの信頼評価というSynoptisの設計は、DEE（分散進化エコシステム）モデル[15]と整合する。DEEは異質なエージェント間の調整が強制的な合意ではなく観察とローカルな解釈から創発しうることを提案する。DEEが哲学的フレームワークを明示する一方、Synoptisは敵対的環境において検証可能な観察を可能にする暗号インフラを提供する。

### 6.4 将来の課題

- モデル検査（TLA+またはProVerif）によるチャレンジ・レスポンスプロトコルの**形式検証**。
- 合理的攻撃者下での信頼スコア均衡の**ゲーム理論的分析**。
- 完全なログレプリケーションなしのクロスエージェントログ検証のための**分散Merkleツリー**。
- プロダクションデプロイメントにおける観察された攻撃パターンに基づく**適応的重み調整**。

## 7. 結論

MCPエージェントネットワークのための相互アテステーションプロトコルSynoptisを提示した。これは38本のサーベイ論文にわたって特定されたギャップ——MCPエージェントにおける双方向暗号アテステーション、動的信頼スコア、改ざん検知永続化を組み合わせた既存システムの不在——に対応する。Synoptisは一方向的・中央集権的なAttestMCPアプローチを、ハッシュチェーン不変性を持つ双方向・分散型相互アテステーションに拡張する。5次元信頼スコアモデルは微細で時間変化する信頼評価を提供し、チャレンジ・レスポンスプロトコルは構造化された紛争解決を可能にする。[主要な定量的結果は評価後に挿入予定。] SynoptisはKairosChain MCPサーバフレームワーク内のオープンソースSkillSetとして実装されており、[リポジトリURL]で利用可能である。
