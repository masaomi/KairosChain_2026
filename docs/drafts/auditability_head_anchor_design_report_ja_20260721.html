Auditability Head-Anchor 設計解説
— 内部鎖の全履歴を公開 anchor で証明可能にする（MPR-1〜9）

対象: docs/drafts/auditability_head_anchor_design_v0.3_draft.md（FROZEN v0.3・2026-07-21）
凍結根拠: R3 multi-LLM review 6/6 APPROVE（subprocess）＋ persona 3/3 APPROVE ／ 継承: ANC-1..9・AHM-1..9・BRD-1..4 ／ 種: L2 handoff（auditability 改良4レベルの L1）

1. なにが問題だったか — 現行 anchor が証せない三つのこと

企業が社内の閉じた（内容非公開の）KairosChain で計算を記録し、公開サーバーへ anchor して 「確かに計算し、記録し、改竄していない」を第三者に示したい——この使い方を想定すると、 現行の anchor（文書の指紋ひとつを載せるだけ）には次の穴があります。

証したいこと現行理由
一員性 — その指紋が内部鎖のメンバーであること✗ anchor は指紋単体しか見ない。鎖と無関係な単発の指紋でも見た目が同じ
不改竄 — 内部鎖が書き換えられていないこと✗ 内部鎖は非公開・単一運営者。外からは検査できない
位置と順序 — 追記の何番目か、どちらが先か✗ 位置の情報が公開されない

2. 中心のアイデア — 束縛対象の格上げ

一言でいうと anchor に「文書の指紋」ではなく「内部鎖の全履歴への封印（累積 commitment）」を載せる。 監査人には鎖そのものを見せる代わりに、証明書（包含証明・consistency 証明）だけを渡す。 内容は一切開示されない。

3. 包含証明 — 「第 N 番の一員」を中身を見せずに証す

封印（root）は Merkle 木で作られます。監査人に渡すのは、当該記録の指紋から root まで登る経路上の 「隣のハッシュ」の列＋位置情報だけ。隣の記録の中身は一切渡りません（MPR-2）。

4. consistency 証明 — 「鎖は書き換えられていない」を支える一本柱

包含証明だけでは、運営者が「履歴を作り直して新しい封印を anchor する」攻撃を検出できません。 一つひとつの証明は新しい封印に対して正しく検証できてしまうからです。 そこで MPR-9: 同じ鎖 identity を持つ二つの封印は「後が前のそのままの延長である」ことを consistency 証明で検証できなければならない、と定めます。

5. 三者の構図 — なにが公開され、なにが closed のままか

6. 9つの invariant（MPR-1〜9）早見表

ID名前一言でいうと
MPR-1束縛の commit と一貫性 head binding は entry_hash の中（差し替えは発覚する）。新規 entry のみ付与で AHM-4 保全。検証できない要素は「情報」と正直に表示し、証明の信用を借りない
MPR-2隣の封じ込め 証明が運ぶのはハッシュと構造情報のみ。中身は越境しない（低エントロピー時の残余は開示）
MPR-3自己記述の決定性 検証の作法（規約）に名前を付けて commit し、定義自体も検証可能に。葉と中間節の取り違え偽造を性質として排除
MPR-4監査人の自足性＋信頼基盤の開示 検証は5入力で完結・鎖アクセス不要。ただし「作るのは運営者」「commitment≠中身」の線引きを内蔵
MPR-5存在は証す・網羅は証さない anchor しなかった計算は見えない。結果非依存で anchor する規律は L2 の仕事と明記（正直な限界）
MPR-6証明の射程 一員性・不改竄・位置・順序まで。品質・正しさは語らない（それは L3/L4）
MPR-7最初の anchor 以前は時刻の裏付けなし 古い記録も一員性は証明できる。証明できないのは記録時刻。anchor が密になるほど以後の窓は狭まる
MPR-8位置の commit と順序 証明は位置と列の長さを commit。「第 N 番」が言え、順序は位置の引き算。時計は使わない
MPR-9anchor 間の整合 同一 identity の封印同士は延長関係を検証可能に。非提出は「未確立・報告対象」（有罪ではない）

7. 成長経路 — この設計が「やらない」と決めて次段階へ送ったもの

正直な限界（設計に内蔵された開示）
  • anchor log 自体を丸ごと差し替える攻撃は L1 では検出できない（→ L2 が塞ぐ）
  • 「都合の悪い計算を最初から anchor しない」は監査人に見えない（→ 結果非依存の規律は L2）
  • 証明は計算の「正しさ」を語らない（→ 再実行は L3、検証可能計算は L4）
  • 最初の anchor より前の記録は、一員性はあるが時刻の裏付けがない（MPR-7）

7-補. 将来ステップ L2/L3/L4 の詳細と進め方の方針（2026-07-21 追記）

前節の階段（図5）の各段について、「どの穴を・どう塞ぐか」をもう一段噛み砕いて記録する。 前提として押さえるべき関係: L1 は証明の「形」を完成させた。L2〜L4 は同じ証明が「信じられる条件」を 積み増すだけで、L1 の証明機構（封印・包含・consistency・位置）は以後どの段でも作り直さない（凍結 §5）。

L2 — head の外部公開と cadence（ANC-4/6）

  • 塞ぐ穴: ①anchor log 自体の丸ごと差し替え — 公開 anchor log も単一運営者のサーバー上のファイルなので、 内部鎖を書き換えたうえで anchor log ごと整合するよう作り直せば、監査人には「最初からそうだった」ようにしか見えない （ANC-1 が scope X の正直な残余として開示・MPR-4/9 が「L2 で閉じる」と明記）。 ②見せ分け（split-view） — 監査人ごとに別の log を見せる攻撃。 ③選択的 anchor — 都合の悪い計算を最初から anchor しない自由（MPR-5 の開示済み限界）。
  • どう塞ぐか: anchor log の head を運営者の支配が及ばない場所へ定期的に刻む（公開 git・Zenodo・ 他の KairosChain インスタンス（相互係留）・RFC 3161 時刻認証局 等）。log を作り直した瞬間、 再計算 head ≠ 外部に残る head で露見（ANC-4）。見せ分けは独立した読者同士の head 突き合わせで検出（ANC-6）。
  • cadence が本質: 一度でなく律動として刻み続ける。刻みが密なほど、記録を後から挟み込める時間の窓が狭まる （constitutive note §7 の姿勢）。結果非依存で anchor する規律の機構化もこの段の仕事。 その際、凍結 ANC v0.5 の human-gate 非目標（自分の著作物の anchor は人間の行為）と衝突しない形 （監査用の鎖 head の自動係留と著作物の人間ゲートの区別）で設計する（R1 の教訓・handoff 記録済み）。
  • 無限後退の閉じ方: 「anchor サーバーを誰が anchor するのか」は、より深い権威でなく インスタンス同士が互いの head を普通の anchor entry として刻み合う相互係留で閉じる。 鎖の head も「ただの digest」なので L1 の証明の形は不変。

L3 — 独立監査人による再実行と foreign 署名（reproduces）

  • 塞ぐ穴: L1+L2 でも証せるのは「記録した・書き換えていない・この順序」まで。 計算の中身の正しさは語らない（MPR-6 の線引き）。さらに現状の anchor は same_party＝運営者の自己申告（ANC-8）。
  • 第一段 — 材料の by-reference 記録: 計算の {入力・コード・実行環境イメージ・出力} を、 中身は鎖に載せず指紋だけで内部鎖に記録（BRD-2 の作法）。 「再現性」が検証可能な主張になる: 記録どおり再実行して出力の指紋が一致するかを見ればよい。
  • 第二段 — 第三者の署名: NDA を結んだ独立監査人が closed 環境で再実行・一致確認し、 公開 anchor に reproduces attestation を自分の identity で署名して載せる。 ANC-8 の relation 導出により、監査人の署名は自動的に foreign（第三者の裏書き）と表示される。 自己申告が第三者の証言に変わる瞬間。
  • 限界: 監査人の独立性は identity 発行に依存（ANC-8 開示済み）。再実行で検証できるのは決定的な計算のみ。 実行環境の同一性の担保が実務上の急所。それでも L4 と比べ桁違いに現実的＝当面の現実解。

L4 — ゼロ知識証明（中長期フロンティア）

  • 塞ぐ穴: L1〜L3 でも「どの記録か」（指紋・位置・存在のパターン）は見える。 競争環境ではこれ自体が漏らしたくない情報になり得る。
  • どう塞ぐか: zk-SNARK/STARK で「封印 root の中のある葉を知っている」（どの葉かは明かさない一員性証明）や、 「commitment C_in の入力に f を適用すると commitment C_out になる」（計算実行そのものの証明＝MPR-6 が除外した 「正しさ」の再実行なし公開検証）を、対象を明かさず誰でも検証できる形にする。
  • 今やらない理由: 証明生成コストが対象の計算より桁違いに重く、ゲノムのパイプライン級には非実用（現状）。 L1 の封印の作り方は L4 でもそのまま使えるため、待つことに設計上の損はない。
進め方の方針（masaomi 決定・2026-07-21）
  • L3（計算の正しさ・再現性・監査性の検証）は GenomicsChain 側で推進する。 こちら（KairosChain 本体）は L1 実装 → L2 の順でこのまま進める。
  • L2 の信頼は、別の運営者が管理する HestiaChain インスタンスの存在によって積む。 head の刻み先が「運営者の支配の外」であることが ANC-4 の要件であり、 独立運営のインスタンスとの相互係留（federation）はその最も KairosChain らしい実現形。 ANC v0.5 §5 の想定どおり、新しい種は不要——相手インスタンスは「ただの foreign depositor」として振る舞う。
段階塞ぐ穴手段信じられるようになる主張状態・方針
L1一員性・不改竄・順序が証せない累積封印＋包含/consistency 証明 「記録は鎖の第 N 番。anchor 間で書き換えなし」FROZEN v0.3・実装へ
L2anchor log ごとの差し替え・見せ分け・選択的 anchorhead の外部刻み＋cadence＋読者間突合（ANC-4/6） 「その anchor log 自体が本物で、律動的に刻まれている」設計未着手。別運営者の HestiaChain インスタンスとの相互係留で信頼を積む方針
L3正しさ・再現性が自己申告どまりby-reference 記録＋独立監査人の再実行＋foreign 署名 「第三者が再実行して一致を確認した」draft の種あり。GenomicsChain 側で推進
L4どの記録かを明かさないと証せないゼロ知識証明・検証可能計算 「対象を明かさず一員性/正しさを公開検証」中長期・待って損なし
