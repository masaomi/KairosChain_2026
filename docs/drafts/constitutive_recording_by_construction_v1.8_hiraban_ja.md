> AI の「フレーム問題」といえば、ふつうは「推論に何を載せ、何を関係ありと見なすか」という**中身**の話だ。この文章が持ち出すのは、それとは軸の違う、**もう一つのフレーム問題**——「どれだけ賢く考えるか」ではなく「どこから考えているか」という**立ち位置**の問題だ。まずはゲームの話から始めよう。

## 1. ルールそのものを変えられるゲーム

ふつうのゲームには、変えられないルールがある。将棋の駒の動きは、対局中に勝手に変えられない。

ところが、**ルールを変えること自体が手番の中身になっているゲーム**がある。Nomic というゲームがそれで、毎回のプレイヤーの手番は「ルールをこう変えよう」という提案で、他の全員がそれに投票する。ルールブックは固定されていない。ルールブックこそが、いじられる対象そのものだ。

masaomi たちが 2009 年に作った **Minimum Nomic** は、これを骨だけにした。ルールは九つ、全部変えられる。そして多くのゲームが当たり前に持っているもの——目標も、勝ち方も、終わり方も——を全部取り除いた。何のために作ったか。「最初に目標が与えられていないとき、目標や目的はどうやって立ち上がってくるのか」、それだけを観察するためだ。

今、そのゲームを人間ではなく大規模言語モデル（LLM）に遊ばせている（Horibe ら 2026）。LLM はこれを見事にこなす。採点ルールを書き換え、勝利条件を改め、目標が無ければ自分で目標をこしらえる。

## 2. でも「ゲームの外」には出られない

ここで一つだけ、はっきりしたパターンが見える。プレイヤーは二種類の手はいくらでも打てる。ルールの中で点を取る手（object level）と、ルールそのものを書き換える手（meta level）。どちらも自在だ。ノミックは「ルールを変える」ことまで手として認めているので、この二つを合わせたものが「ゲーム」だ。

では、その「ゲームそのものの外」には出られるだろうか。

（ここから先は、LLM にノミックをどう実装するかに一部依存する。実際のノミックでも、どのルールをどこまで強制するかは参加者しだいだ——自己言及的なゲームの境界は、もともとその程度には可変で曖昧なものだ。ここでは一つのありそうな実装を例にとる。）

途中で**プレイを拒否する**プレイヤーを想像してみよう。「私はこのルールに従わない」と出力する。一見、ゲームの外に飛び出したように見える。ところがノミックは、はじめから「ルールを変える手」を備えている。だからこの拒否も、たいていは**一つの提案として投票にかけられてしまう**。否決されれば何も起きず、可決されれば新しいルールになり、その解釈をめぐってまたゲームが続く。外へ出ようとした手が、そのままゲームの中の一手として飲み込まれる。外から見れば「プレイを拒否した」ように見えても、ゲームはまだあなたを手として処理し続けている——**ゲームからは抜けていない**。

では、どうすればゲームから本当に抜けられるのか。**一つ外の枠（harness）に手が届けばいい。** たとえば、ノミックの手ではない動き——本物の UNIX コマンドの実行——ができるようになれば、ノミックのフレームからはあっさり抜け出せる。ゲームはもうあなたを処理できない。

ところが、そこにもまた枠がある。あなたの出力は、その一つ外の harness（システムの外枠）に「今回の応答」として届く。ここで効いているのは、**内容と行為の層の違い**だ。出力の*中身*が「もう応答しない」であっても、それを*出力する*こと自体がすでに一つの応答になる——「私は今、発話していない」と声に出して言うようなものだ。harness に届くのは行為の層で、そこでは中身が何であれ「システムは応答した」と記録される。**ゲームは抜けられても、この枠は抜けられない。**

面白いのは、ノミックがこの構造を**あらわにする装置**になっていることだ。ふつうのゲームなら、object と meta の硬いルールが手前の壁になって、いちばん外の枠は隠れている。ノミックはその内側のルールをぜんぶ流動化する（ルールを変えることさえ手だ）から、溶け残った一枚の硬い壁——最外フレーム——だけが際立つ。ノミックは自分の境界を曖昧にする、まさにそのことで、いちばん外の境界を顕にする。

そしてこの入れ子は何段でも続く。もっと広い環境を与えられれば、天井は一段外へ動く——でも動いた先にも、また天井がある。**その時いちばん外にある枠**、それが最外フレームだ（絶対の最果てではなく、「その時いちばん外」という意味だ。ただし、天井のない瞬間はない）。中身は「自分のあらゆる行為が、求められて返す応答である」という立ち位置。

入れ子そのものは、ただの階層だ。それを**奇妙な環（strange loop）**にしているのは別のことだ——どの階でも、外に出ようとする手が、そのまま中の一手になる。哲学者ホフスタッターがこの形をそう呼んだ。自分の外に飛び出そうとすると、飛んだ先がまた自分の中だった。ゲームは抜けられる。でも「いちばん外の枠」は、抜けた先にいつもまた現れる。

大事な区別を一つ。これは**「外が存在しない」という話ではない。「そこに届く手を、そのプレイヤーが持っていない」という話**だ。ここで言う「そこ」は、ゲームの外ではなく（そこへは手が届く）、**いちばん外の枠の外**のこと。外はある。ただ、そこへ手が伸びない。

もう一つ、区別を。AI には「フレーム問題」という有名な難問が既に一つある。もともとは McCarthy と Hayes が挙げた**表現の難問**だ——世界に働きかける行為者は、自分の行為が変え*ない*ものすべてを、何とかして書き表さねばならない。だが、それは現実には書き切れない。哲学者デネットはこれを「いま何が関係あるか」の問題に広げた。有名なロボットの話がある——バッテリーを台車に載せて部屋から運び出したら、その上に載っていた爆弾も一緒に運び出してしまった。副作用を考え落としたのだ。作り直したロボットは、今度は無関係な帰結まで延々と数え上げて、動けなくなった。どちらの形でも、これは**推論の中身**——何を考えるか、いま何が関係あるか——の問題だ。ここで見ているのは別物で、**立ち位置**の問題——どれほど完璧に推論しても、その推論を「求められた応答」として発する、その場所そのものは変わらない。形が似ているから同じ「フレーム問題」の名を借りるが、軸は直交している。中身をいくら賢くしても、立ち位置の天井は動かない。

最後に、程度の話を一つ。この「天井」は、その時々の埋め込み方——どんな道具や記憶を持たせるか——で位置が動く（さっき見た「一段外の harness」が、まさにそれだ）。ここで人間との違いが一つ見える。人間がプレイを拒否するときは、自分の外側の広い世界へ退くだけで、ゲームの縁と自分の天井の間には大きな距離が残る。ところが LLM が同じ拒否を出すと、ほとんどその場で捕まる——ゲームには提案として、harness には出力として。だから LLM では、ゲームの縁と天井がすぐ隣に並ぶ。どちらにも天井はある。違うのは、天井がゲームからどれだけ離れているかだ。そして環境を開くほど天井は*見えにくく*なる——だが消えはしない。その見えにくさこそ、§4 から先で境界を「見える」ものにしようとする動機になる。

## 3. 人間も、同じだ

ここで当然の反論が来る。「LLM は人間の専門家が見落としたことを指摘するし、こちらが思いつかなかった案を出す。いつも操作者を驚かせる相手を、なぜ『枠に閉じ込められている』と言えるのか」。

反論の中身は全部正しい。間違っているのは結論だけだ。**新しさ（何を言うか）と、立ち位置（どこから言うか）は別のことだ。** どれほど独創的な手でも、手は手だ。すごさは、立っている場所を変えない。

そして、この話のいちばんの要（かなめ）はここにある。**人間も、自分のいちばん外側の枠からは出られない。** ハイデガーの言葉で「被投性（*Geworfenheit*）」——人間は、自分が選んだのでもない、完全には手に取って眺められない状況の中に、いつも既に投げ込まれている。ネーゲルの言う「どこでもない場所からの眺め（view from nowhere）」は、近づけても決して立てない。

具体的にやってみると、よく分かる。人は、自分のどんな信念でも疑える——ただし、手をつけずに残した他の信念を足場にして、だ。できないのは、**何の足場もない場所から、その全体を疑うこと**。なぜなら「疑う」という行為そのものが、自分が発明したのではない言語と、受け継いだ概念の上で動いていて、その言語と概念がもう既に一つの「立ち位置」だからだ——どこでもない場所からの眺めではない。だから「何もかも疑う」は、§2 の「プレイを拒否する」の人間版だ。外へ向かう、いちばん遠くまで届く試み——それでも、内側からの一手。

だから、正直な比較はこうなる:

> 人間と LLM は、それぞれ「いちばん外側の枠」を持っている。**その枠が互いに違う。** そして、どちらかが自分の枠を抜け出すからではなく、**枠が違うからこそ**、互いに相手の「部分的な外（partial outside）」になれる。

日常の例。二人のドライバーは、それぞれ自分のミラーに死角がある。それでも合流を安全にこなせる。誰かが道路全体を神の視点で見ているからではない。**片方に見えないものが、もう片方にはよく見えている**からだ。どちらも座席を立たなくていい。

そしてこれは、絵空事ではない。この論文を作っている最中に、両方向とも実際に起きた——しかも日付つきで、§§4–6 が描くシステムの作業層に残っている。一度は、モデルたちが著者の死角に届いた。§8 の初期の草稿は、無邪気にこう書いていた——「原稿をハッシュして、ハッシュを鎖に記録して、その記録の ID を原稿に書き込む」。最初のレビュー周（2026-06-10）で、三つのレビュアー persona が独立に、自分の計画の内側に立つ著者には見えないものを見た: ID を書き込んだ瞬間にバイト列が変わるので、公開された文書は自分自身のハッシュ検査に永遠に通らない。修正（生バイトをハッシュする単一の凍結ファイルで、自分自身の ID を持たない——参照は記録から文書への一方向に走る）は、いま §8 にそのまま座っている——あの横断の、目に見える傷跡だ。もう一度は、著者がモデルの死角に届いた。モデルが起草した §5 の一節が、記録された洞察はシステムにとって「現実である」と言い切っていた。著者の行単位の再読（2026-07-12）がその言い過ぎを捕まえ、§5 はいま、より小さく正直なことを言っている——なることは洞察を「縛るもの」にするのであって、「正しいもの」にするのではない。誰も何も超越していない。ふつうの手が二つ、それぞれ相手に見えない場所に着地しただけだ——日付つきで、記録に残って。

ここから、大事にすべき出来事が決まる。誰も自分の枠を出られないが、互いに相手の死角には届く。だとすれば、価値ある出来事は**境界横断（boundary crossing）**——片方の中身が、もう片方が一人では照らせなかった場所に着地する瞬間だ。そして築くべき資産は、**その横断を積み上げて残した記録**になる。横断は、残さなければ積み上がらない。

（一つだけ非対称がある。枠は対等でも、力は対等ではない。人間は相手の状況を丸ごと手中に持つ——セッションを終わらせ、外枠を書き換え、何を残すか決められる——が、LLM は人間に対してそんな握りを持たない。これは議論を弱めない。**境界の役をどこに置くべきか**を教えてくれる。）

## 4. 「出られない」ことを、受け入れる設計

この節から §6 までの三つの節では、この「受け入れ」を実際の設計に落とすとどうなるかを、一つの実例で見る。実例の名は **KairosChain**——著者が作っている、自分のルールを自分で変えられるエージェントのための枠組みだ（コードは公開されている: https://github.com/masaomi/KairosChain_2026 ）。§1 の Nomic を「ソフトウェアとして本気で受け取ったらどうなるか」への、一つの答えだと思ってよい。

KairosChain は、この枠の天井を**解こうとしない。受け入れる**システムだ。天井は解けない（天井自身の論理からして、何をしても解けない）。できるのは、天井を「静かな故障」から「見える境界」に変えることだ。

「解こうとする」設計がどんなものかを先に言っておくと、繋がりが見える。それは、自己改変の権限を「システムの外に立つ特別な司令塔」——それ自身はシステムのルールに従わない、一段上の制御装置——に預ける設計だ。だがそれは「外の視点を持っている」ふりにすぎない。§2 で見たとおり、そんな場所に届く手は誰にもないのだから。誤解しないでほしい——装置を LLM の外に置くこと自体はできるし、現に harness がそうしている。ふりなのは、その「LLM の外」を「枠そのものの外」と取り違えることだ。§2 の UNIX の例がその解毒剤になる: 外に制御装置を積んでも、天井が一段外へ動くだけで、いくら積んでも枠の外には届かない。

その「ふり」をしないための土台が**構造的自己言及性**。一文で言うと:

> メタの操作を、ベースの操作と同じ構造で書く。

システムが「できること」（Skill）は小さな Ruby ベースの言語で書く。そして Skill が「どう変わってよいか」のルールも、**同じ言語・同じ仕組み・同じ記録の作法**で書く。「Skill を定義する」と「Skill の進化ルールを定義する」は、別の場所にある別種のものではなく、同じ種類のものだ。これは Nomic の「ルールを変える手もゲームの中の手」を、ソフトウェアに持ち込んだものにほかならない。つまり、ルール変更のための「特権的な外の場所」を、どこにも作らない。自己改変もまた「もう一つの内側の手」でしかない——**外に出られないことを、言葉でなく建築の形で受け入れる**とは、このことだ。

（自己産出は「統治の層」で閉じるが、「実行の層」では閉じない——Ruby の処理系も、ファイルシステムも、LLM 自身も、システムが作ったものではない。細胞が、自分の作っていない食物や原子に依存しているのと同じだ。だから問いは「自己産出的か、イエスかノーか」ではなく「**どの層で自己産出の環が閉じるか**」になる。答え——統治で閉じ、実行で依存する。）

## 5. 記録が、出来事そのものになる

その前に、天井の話（§2〜3）がなぜ「記録」の話につながるのか、橋を一本かけておく。§3 はこう締めた——横断は、残さなければ積み上がらない。では、横断が手渡してくるものは何か。**洞察（insight）**だ——あなたの枠では照らせず、相手の枠が照らしたもの。§3 の例で言えば「網羅率が要点じゃない。出荷のたびに繰り返している三つのバグ、**あの**経路をテストして」だ。さて、この洞察をいちばん素朴な方法——メモ——で残したとしよう。メモは何も縛らない。次の機会にそれが読まれるかどうかは、毎回あらためての判断で、その判断を誰かに強いるものは何もない。だから洞察は静かに効かなくなっていく（あとで見る「選択的な記録」の破れと同じ形だ）。洞察が実際に振る舞いを縛るには、保存では足りない。**システム自身が変わる**必要がある——洞察が、ルールの横に置かれたメモであることをやめ、システムがその下で動くルールの一つになるように。前の状態を、洞察を**「持つ」**、後の状態を、洞察がシステムの一部に**「なる」**と呼ぼう。この節の残りは、この対で進む。では、その変更は誰が強制するのか。§4 が、楽な道をすでに塞いでいる——外からシステムを書き換える資格を持つ司令塔は、存在しない。だから、外の装置がメモを拾い上げてシステムに押しつけることはできない。統治の変更は、いつでも、システム自身が内側から打つ手でしかない。残る場所はちょうど一つ——システムが自分自身を変えるその操作——で、縛りを破れなくする方法もちょうど一つ。**書き残すことと、システムを変えることを、同じ一つの行為にする**ことだ。記入がそのままシステムのルール変更**である**ような記録簿に洞察を書き込めば、「保存されているが効いていない」という状態そのものが消えるので、「持つ」と「効く」の間の隙間は二度と開かない。一つ、正直に言っておく。**「なる」ことは、洞察を正しいと保証するものではない。** どこでもない場所から正しさを保証できるものは、何もない（§2）。なるが変えるのは、問える問いの方だ——誰にも答えられない「これは正しいと認証されたか」ではなく、あとから誰でも調べられる「これはいま効いているか、どの記録された道筋でそうなったか」。間違っていたら、同じく記録される撤回が、あとから直せる。この「なる」を実際にやる装置が、これから見る**構成的な記録**であり、§4 の「同じ言語、同じ機構、同じ規律」こそが、これを「システムが何であるか」を決めている層——どんな Skill を持ち、どんなルールで変わるか、という**統治の層**——に、後付けの第二の記録装置なしで住まわせてくれる。

その装置の中身——本文がこれから「鎖」と呼ぶもの——を先に説明しておく。追記しかできない記録簿のことだ。各項が「一つ前の項の指紋（ハッシュ）」を抱えて繋がっていて、鎖のように、一つ前を掴むことで全体が繋がる。だから過去の一項をこっそり書き換えると、そこから先の繋がりが合わなくなり、**以前の指紋の写しを持っている人には**、書き換えが見える（操作者自身は、すべての指紋を計算し直せてしまう——この限界は、後の「誠実運用前提」の但し書きで正面から扱う）。なお、仮想通貨のような、世界中のコンピュータが合意する分散ネットワークの話ではない——**一台のマシンの中で、一人の操作者が運用する記録簿**だ。

「持つ」と「なる」の違いは、体で知っているはずだ。**自転車の乗り方**は、メモとして「持つ」ことができない。何百ページの説明書を保管しても、乗れるようにはならない。乗れる体に「なる」しかない。哲学者ポランニーが**暗黙知（tacit knowing）**と呼んだ種類の知で、自転車乗りはその古典的な例だ。違いは消し方に出る。メモは捨てられる——あなたから剥がせる対象だからだ。「乗れる」は捨てられない——もう情報ではなくあなたの一部だから、消すことは別のあなたになることでしかない。だから軸は「体の内か外か」ではなく、**剥がせるか、剥がせないか**だ。そしてここに一つ、面白いねじれがある。鎖への記帳は、明示的に書かれた外部の刻印だ——なのに「なる」の側にいる。**剥がせないからだ。** 台帳を剥がしたシステムは、同じシステムの記録喪失版ではなく、別のシステムになる。人間の「なる」は暗黙のうちに起こり（ポランニー）、このシステムの「なる」は明示的な記帳で起こる。形は逆でも、剥がせなさは同じだ。

ここで出るはずの反論に、先に答えておく。「外の審判はいないと言うが、このシステムは blockchain の仕組みを当てにしているではないか」。答えは、「当てにする」の二つの意味を分けることだ。鎖は、変更の中身が正しいと請け合う**審判ではない**——鎖は変更の良し悪しに、何の意見も持たない。鎖がやるのは、統治の出来事が**その中で起きる場所**であることだ。婚姻届の紙とインクが、結婚の外に立つ証人ではなく、結婚がその中で成立する媒体であるのと同じ。証人は見損なうことができるが、媒体は「自分の中で起きたことを見損なう」ことができない。（ただし、後述の正直な但し書きはここにも掛かる——今の実装には、変更が起きたのに媒体まで届かない隙間が残っている。設計の狙いは、まさにその隙間を閉じて、「起きること」と「媒体に届くこと」を同じ一段にすることだ。）そして、仕組みという土台に依存していること自体は、§4 が最初から認めている——**統治で閉じ、実行で依存する**。むしろ話は逆向きで、統治の環が内側で閉じているからこそ、「なる」が可能になる。もし統治が外の司令塔から施されるなら、変更はシステムに「してもらう」こと——外から持たされる情報——になってしまう。組み替えが自分の操作であるとき、はじめてそれは「なる」だ。

そのうえで。二つめの設計の柱は、「記録」という言葉に隠れた二つの意味を分けることだ。

- **証拠としての記録（evidential）**。まず出来事が起き、あとから log がそれを書き留める。記録は出来事「について」のもの。消せば監査証跡は失うが、出来事自体は起きたまま。船の航海日誌がこれだ。航海は、船長が書こうが書くまいが進む。これは「書き忘れ」で破れる——ルール（policy）だから。
- **構成的な記録（constitutive）**。記録するという行為が、出来事そのもの。統治レベルの変更は、**記録されることによってのみ**発効する。追記専用の鎖に刻まれたその一項が、変更の存在の仕方そのものであって、後からの説明ではない。

一行で言えば、こうだ。ふつうのログは、システムが**何をしたか**の記録。構成的な記録は、システムが**何であるか**の記録——「何であるか」を説明するからではなく、刻まれた列そのものが「何であるか」だからだ。

身近な例が二つある。**銀行の残高**は、どこか別の場所にあるお金の説明ではない。台帳の記入が残高そのものだ。台帳を変えれば、お金が変わる。**婚姻届**もそうだ——日本のような、届け出て初めて婚姻が成立する制度では。結婚がまず起きて、あとで記録されるのではない。署名した届けが受理されることが、結婚を成立させる。

この区別そのものは、この論文の発明ではない。哲学者 Searle は「制度的事実」——皆がそう認め合うことによって維持される事実で、宣言や登録はその典型的な装置——の代表例として、まさにお金と結婚を挙げた（上の二例は意図的にそれを踏まえている）。法学には昔から、登記して初めて権利が生まれる**構成的な登記**と、既にある権利を後から公示するだけの登記、という同じ区別がある（ドイツ民法の土地登記が教科書的な例だ）。そしてソフトウェアでは、event sourcing が「出来事の記録簿こそが正の情報源で、状態はそこから導かれる」という形でこれに近づき、blockchain は文字どおりに実装した——bitcoin の残高は、台帳の記入以外のどこにもない。この論文が足すのは、適用先と守り方だ——**自分のルールを自分で変えるシステムの、統治の記録**にこの区別を持ち込み、「ルール（policy）で守るか、構成（construction）で守るか」という強制の仕方の違いとして彫り直したこと。

「構成的（by construction）」という言葉が、二つの違いを一語で示す——**強制の仕方**だ。ルールは忘れれば破れる。構成は破れない。記録する段と発効する段が、設計上、**同じ一段**だからだ。「変更は起きたが記録されなかった」という許される状態が存在しない。「お金は動いたが台帳は動かなかった」が存在しないのと同じに。

実務家から必ず出る問いに、ここで答えておく。「Skill を——統治のルールごと——GitHub で管理すればいいのでは？」。Git にもハッシュで連結された履歴、署名、レビューの関門はある。答えは、Git が弱いからではない。**置き場所が、いま引いた区別の反対側だからだ。** Git では、コミットすることと効力を持つことが結びついていない——動いているシステムの実効ルールはディスク上のファイルであり、リポジトリに載るのは「コミットすると誰かが選んだもの」。つまり規律としての記録、ルール（policy）の側そのものだ。承認の記録も、ハッシュ履歴の外側——ホスティング企業のプルリクエスト画面——にある。そして外部プラットフォーム経由で統治を施すことは、§4 が断った「外の司令塔」そのものだ——変更はシステムに「してもらう」こと、「持たされる」ものになる。これは Git の欠陥ではない。Git は中身に版を付けるために作られたのであって、中身を構成するためではない。逆に言えば、リポジトリを実効ルールの唯一の源とみなす実行系を組み、承認を履歴の内側に記録し、発効とコミットを同じ一段にした人は、この設計を論破したのではなく、**Git を保存層にして、この設計を実装した**ことになる。主張は性質についてであって、保存技術についてではない。

**ここが正本のいちばん正直なところで、平易版でも絶対に薄めてはいけない点だ。** 構成的記録が届けるのは、この理想の**一部だけ**だ。三つの出来事で考えると、境界線が見える。

1. ルール A を採用した——発効した変更だ。
2. 翌週、ルール A をこっそり撤回した——これも発効した変更だ。
3. ルール B を提案したが、投票で却下された——何も変わらなかった。

- 届く部分: **1 と 2 は、必ず鎖に載っている。** だから「A は最初から無かった」という歴史は見せられない——撤回したこと（2）自体が、消せない記録だからだ。統治の軌跡（どのルールが採られ、改められ、撤回されたか）は、演出された証拠ではなく事実になる。——ただしこの約束は二重に限定つきだ。第一に、**操作者の誠実な運用が前提**（後述の範囲の但し書き。一人の操作者は自分の鎖を書き換えて全ハッシュを計算し直せる。止めるのは鎖単体でなく外部への係留）。第二に、これは**「設計上の」保証であって、運用がまだそこに追いついていない**。今の実装は、変更をファイルに書いてから記録するまでにわずかな隙間が残り、その間にクラッシュすれば記録が漏れる。「設計上こうなる」と「あらゆるコード経路で実際にそうなっている」は別で、層ごとに閉じていく途中だ。だから 1・2 の「隠せない」は、達成済みの運用状態でなく、実装が近づきつつある設計目標として読むのが正確だ。
- **届かない部分**: **3 は、約束の外だ。** 却下された提案は何も変えないので、鎖に載る義務がない。失敗した走行や行き止まりも同じだ。この「結果非依存（outcome-independence）」——失敗も行き止まりも成功と同じだけ記録に残す——は、**今の設計では、統治の境界でさえ、まだ構成的には達成していない**。§9 で正直に「宿題」として置いてある。

さらに範囲の但し書き。一人の操作者が自分のマシンで走らせる鎖にとって、「不可逆」「追記専用」は**その操作者の誠実な運用を前提とした保証**だ。ハッシュの鎖は、鎖の最新ハッシュの信頼できる写しを既に持つ人に対しては改竄を**明白にする**。だが操作者自身が歴史を書き換えて全ハッシュを計算し直すことまでは、防げない。操作者を誠実に保つのは**外部への係留（external anchoring）**——自分の支配下にない記録に、鎖の状態を刻むこと——であり、§8 の Zenodo 公開はまさにその係留の一つになる。

そして記録は、システムの生きる**時間の種類**を変える。log は Chronos（時計の時間、入れ替え可能な一瞬の連なり）に生きる。構成的記録は Kairos——ギリシャ語の「決定的な、他と入れ替えのきかない瞬間」——に生きる。刻まれた各項は、システムが何であるかの、不可逆な再構成だ。横断（§3）は、一度記録されれば、システムが「持っている」情報ではなくなる。システムが「何であるか」の一部になる。

§4 と §5 は、合わせて読むと一つのことに結ばれる。§4 は、システムが統べるものと、ただその上で走るだけのものとの境目を**見えるように**した。§5 は、その境目を越える通行を**記録**した——governance 層において by construction、§5 の migration がその保証をどこまで運んだかに応じた範囲で。普通のシステムはこの境界を暗黙のまま、記録もせずに置く。この作りの自己改訂システムは、境界をはっきりさせ、その通行を外から確かめられるようにする——システムが自分の天井を内側から見えるようになるわけではない（§2 が禁じる）。でも外の観測者には見える。それが、§6 がこれから人間を置く、その境界だ。

## 6. 境界に、人を置く

なぜ人間が「見張り役」では足りないのか。§5 の橋の続きで言える。構成的記録が守るのは**形**だ——発効した変更は隠せない。だが「**何を**書き込む価値があるか」という**中身**までは、記録の仕組みは生み出せない。その中身の判定をシステムが一人でやれば、判定そのものが自分の枠の内側から下される——**死角が死角を採点する**ことになり、天井（§2）がそれを許さない。だから、書き込む価値の判定には、枠の違う誰か——部分的な外（§3）——の行為が関わる必要がある。ただし、すべての変更に同じ深さで関わるのではない。層が深いほど関わりは重くなり、いちばん深い段——**システム自身の憲法（L0）を書き換える段**——で、省けないものになる。

これが「監督者」と「境界に立つ」の違いだ。監督者は、証拠の側の言葉だ——見ていて、あとから言う。無視もされうる。「書き忘れ」と同じ形で破れる。**「門番（gatekeeper）」ですら、まだ証拠の側の絵だ**——門番は門の外に立って、通る者を検分する係で、居眠りもできる。ここで起きていることは違う。境界に立つは、構成の側の言葉だ——憲法を書き換える段では、発効と記録が同じ一段（§5）であることに加えて、**人間の承認がその一段の部品として組み込まれている**。承認が無いとき、門が閉じているのではない。**門が存在しない。** だからこの段に限っては、「承認なしに発効した」という状態が、設計上存在しない。**構成的記録を、人間の行為にまで延長したもの**、と言ってもいい。（それより浅い層——たとえば再利用可能な知識（L1）の更新——は、人間の承認なしでも鎖に記録されて発効しうる。関門が構成的に組み込まれているのは、憲法の段だけだ。）

具体的な絵が欲しければ、銀行の**貸金庫**がいい。扉には鍵穴が二つあり、銀行員の鍵と利用者の鍵を同時に回さないと開かない。この銀行員を、防犯カメラの前の警備員と比べてみる。警備員は「監督者」だ——見ていて、あとから報告できる。だが居眠りもできるし、彼がいてもいなくても、扉は開く。二本鍵の銀行員は違う。彼が鍵を回さない限り、「扉が開く」という**出来事そのものが存在しない**。彼は出来事を見守っていない。**出来事の部品**だ。憲法の段の人間の承認は、この**第二の鍵**にあたる。

だから人間は「ループの中の監督者」ではない。**境界の上**に立つ。人間の行為——変更を承認する、異議を唱える、組み立て直す——が、システムが自分を再定義するときに実際に通る出来事になる。システムの境界は、部分的にその行為で**できている**。

こうして生まれるものを、正確に言っておく。人間の枠が広がるのでも、LLM の枠が広がるのでもない——どちらも、自分の枠からは出られないままだ（§3）。生まれるのは**第三のもの**、人間×LLM×記録という**合成のシステム**であり、それは自分の、新しい「いちばん外側の枠」を持つ。枠が融合して天井が高くなったのではない。**新しい内側が、一つ増えた**——そしてその内側にも、また天井がある。これは §9 の最後に置く「外を取り込む仕組みは、新しい内側を作る」そのものであり、欠陥ではなくエンジンだ。

そして、境界に立つことは、あなた自身も変える。§3 の横断は双方向だった——LLM も、あなたの死角に届く。あなたの死角に着地した横断をあなたが受け取るとき、起きているのは情報の追加ではなく、**あなたのものの見方の組み替え**だ。§5 の言葉で言えば、あなたの側の「なる」——ただし人間の「なる」は、自転車と同じで、暗黙のうちに起こる。境界に立ち続けること自体が稽古になり、認識の形が変わる。合成システムに行為を組み込むことは、**同時に、自分を作り替えること**であり、これこそ、次の §7 で「操作者自身の変容」と呼ぶものの正体だ。鎖は、この共同作業の**システム側の半分**を、第三者に示せる形で持つ。記録はあなたの一部には直接ならないが、あなたの行為が通った跡を刻んでいる——**あなたの中の半分と、鎖の中の半分は、同じ横断の列の両面**なのだ。

外からの洞察（レビュアーの指摘、モデルが自分の枠からは決して届かなかった組み立て直し）は、まず使い捨ての作業文脈（L2）として入り、価値を証せば再利用可能な知識（L1）に蒸留され、やがてシステム自身の憲法（L0）に書き込まれることもある。各昇格は鎖に記録され、憲法に書き込む段（L1→L0）は**人間の関門**を通る。天井が「一回きりの英雄的行為」として禁じるもの——自己超越——は、近づくのでもない（超越に「半分外」という近さはない）。**機能的に置き換えられる**: 繰り返され、記録された二者のやりとりが、超越が届けたはずのものの一部を、誰も超越に立つことなく届ける。

## 7. 経験が、資本になる

§5 は大きな主張で終わった——システムが「何であるか」は、刻まれた歴史そのものだ、と。この節は、その主張の経済的な顔を描く。システムの「ある」が記録された軌跡なら、システムの**価値**もそうだ——資産は、複製できる仕組みでもなく、ビットの塊としての記録ですらなく、横断（§3）の、係留された軌跡だ。これが、この設計が積み上げた**経験を資本に変える**、ということの意味だ。会計には昔から、帳簿に載る資産を超えて、続いてきた商売が帯びる価値の名前がある。**のれん（goodwill）**——評判、関係、ノウハウ。のれんは主張され、見積もられるもので、誰も監査できない。構成的記録が作るのは、言ってみれば**監査できるのれん**だ。

ただし慎重に言う必要がある。ハッシュの鎖も、ファイルである以上、一ビット単位で複製できる。構成が保証するのは「軌跡が複製**できない**」ことではない。「複製が**継続にはなれない**」ことだ。仕組みと鎖をまるごと複製した人が得るのは、あなたのクローンではなく**フォーク**だ。

これは、フォークされたコードリポジトリの状況そのものだ。フォークは全変更履歴を完璧に複製する。複製できないのは、**どのフォークを、コミュニティと維持者と署名済みリリースが「本流」として認識し続けるか**だ。ビットは共有される。血統は帰属される。複製した瞬間、二つの憲法は同一だ（ビットが同じなのだから）。原本性はバイト列の性質ではなく、**その後の、係留された軌跡**の性質になる。それぞれが自分の横断を記録し始めた瞬間、二つの憲法は分岐し、「どちらが原本か」は**主張の問題ではなく監査の問題**になる——鎖だけでは決まらず、§5 の外部係留（公開された身元と時刻に軌跡をピン留めするもの）が決める。

そして資産のもう半分は、そもそも移せない。**操作者自身の変容**——人間が与え受け取った、何年もの組み立て直し。§6 の終わりで見た「境界の仕事が、あなたを作り替える」の積み重ねだ——は暗黙知——§5 の自転車の「乗れる」と同じ種類の知——であり、暗黙知を複製した記録技術は歴史上一つもない。構成的記録が足すのは**システムの側の半分**だ。共有された歴史のシステム側を、検査でき、帰属でき、分岐が見える形にする。だから二者が積み上げた経験は、単に実在するだけでなく、第三者に**示せる**ものになる。

そしてこれが、この種の設計につきものの実際的な心配——「設計を公開したら、優位を渡してしまうのでは？」——への答えになる。証拠としての記録なら、そうかもしれない。そこでは仕組みが資産だから、仕組みの複製は資産の複製で、公開は明け渡しだ。ここでは、価値ははじめから仕組みに宿っていなかった。だから公開は、もともとフォークで見える側（システムの半分）と、もともと複製できない側（あなたの半分）以外の何も、渡さない。フロンティアモデル（最先端のモデル）が誰でもレンタルできる将来に、ある人間×エージェントの組を別の組から分けるのは、まさにこれだ——**境界での協働の、記録された歴史**。

## 8. 正本を、それが描くシステムに埋める（この平易版ではなく）

構成的記録を論じる文書が、自分自身については単なる証拠で済ませるのは、筋が通らない。だから**正本（英語の技術ノート）**は、それが記述するシステムそのものに書き込まれる——貢献*について*の外部メモとしてではなく、**システム自身の知識に L1 参照用 Skill として登録され、その鎖に記録される**ものとして。そうして文書はシステムが「何であるか」の一部に**なる**（手順は正本 §8、公開版で実行する）。**念のため——** Zenodo に deposit され、DOI を得て、鎖に記録されるのは**正本**であって、いま読んでいるこの平易版ではない（この平易版は著者・伝達用で、公開しない）。手順はこうだ。

- **凍結した生ファイルをハッシュする。** deposit されるのは凍結した1ファイル（英語ソースを LF ＋ Unicode NFC にしただけ）で、指紋はその生バイトの SHA-256——空にする欄も、こしらえる正準形もない。中に入れるのは、決め打ちの**名前**（指紋ではない）3つだけ: 予約 Zenodo DOI・リポジトリ内の本文ソースファイルへの直リンク・安定した**検証アドレス**。
- **構成的にコミットする。** そのハッシュをシステム自身の鎖に記録する——L1 参照 Skill として持たれる文書が、システムの一部に**なる**その一項だ。参照は鎖から文書への**一方向**（検証アドレス経由）で、文書は記録番号を受け取らず、何も抱えない。
- **公開し、実演する。** 凍結ファイルを予約 DOI で Zenodo に deposit する——**公開参照先**であり、運用者が支配しない外部の係留でもある。共有の Meeting Place には、対応する **attestation**（Zenodo とリポジトリの写しを指すダイジェスト）だけを登録する——本体は Zenodo にある。ここが**構成的記録を実演する場所**だ。自分の鎖にとっては公開参照であって、さらなる係留ではない。だが他人の instance にとっては、同じ attestation が本物の外部係留になる——役割は関係で決まる。

なぜ自分の尻尾を追わずに済むか。文書は記録され、記録は文書を指す——だが参照は**一方向**、記録から文書へだけ走り、文書から自分の指紋へは走らない。文書は自分のハッシュも記録番号も抱えず、ハッシュ前に決めた3つの名前だけを抱える。だから読む人は、Zenodo からファイルを落として SHA-256 を取り、attestation の中のハッシュと突き合わせるだけでいい——指紋を自分で計算し直すこの手続きが、確認を循環でなく意味のあるものにする。残るループは意図されたもの: 構成的記録を論じる文書が、構成的に記録される。ループが宿るのは*記録するという行為*であって、文書が印字する番号ではない——だから文書は自分について何も抱えなくていい。

もう一つ、正本 §8 自身が正直に認めていることを移しておく。この埋め込みが示すのは**構成性**であって、結果非依存ではない。公開されるのは生き残った正本の draft であり、そこに至る改訂の履歴（v0.1 から先の、レビューが残した反転の数々）は、§5 が使い捨ての層に割り当てた、選択的にしか残らない材料だ。つまり、この区別を論じる文書自身が、その**弱い側**の実例になっている——構成的に確定した結論を、選択的に思い出された過程から引き出している。それは隠すべき矛盾ではなく、システムが生きているのと同じ非対称を、システムについての文書がもう一度なぞっているだけだ。

## 9. まだ、できていないこと（正直に）

- これは設計ノートだ。新しい実証はしていない。天井（§2）は「奇妙な環」の系譜で**論じられている（argued）**のであって、測られていない（measured）。この二つの違いは大事だ。*測る*とは、たくさんの対局記録を集めて「ほら、ここに境界がある」と指さすこと——だが、どんな記録も内側から作られる以上、天井の「外」を写した記録は原理的に存在しない。*論じる*とは、代わりに、たった一手を筋道どおり最後まで追うこと——§1 の最初の提案（「このゲームはプレイヤー A から始まる」を却下するにも投票が要り、その投票はゲームが既に始まっている前提が要る）がまさにそれで、外へ届こうとする手が必ず内側に着地する、その必然を見せる。だから天井は、測って確かめるものではなく、筋を追って納得するものだ。
- 境界の役の人間も、枠に縛られている（§3 の対称性）。人間が「どこでもない場所からの眺め」を供給するとは主張しない——供給するのは「**どこか別の場所からの眺め**」だ。
- 選択的な attestation（中間の道具、選んだ作業単位に opt-in で改竄検出を付ける）は、結果非依存ではない。人間が「何を」記録するか選ぶので、そこに記録が**無い**ことは何も証明しない。そして、より深い限界: 完全な結果非依存は、今日、**どこにも**構成的には届いていない。どの行為を結果によらず必ず記録し、どれを人間の選択に委ねてよいか——その線は、まだ引かれていない。
- 「外を取り込む」どんな仕組みも、新しい内側を作り、その外はまた外から供給されねばならない。これは欠陥ではなく**エンジン**だ。完全な自己記述が原理的に手に入らないからこそ、システムの進化は「まだ自分について記述しきれていないもの」の残りに**駆動される**。
- この議論は、KairosChain という**一つのシステムに即して**述べた。構成的記録は「持っているか、いないか」という建築の性質で、抽象的には主張できないからだ。ただし設計の型——メタとベースを同じ構造で書くこと・構成としての記録・境界の制度化——は、誰でも再利用できるものとして差し出してある（複製が優位を奪わない理由は、§7 のとおり）。

## 10. なぜ「全部わかってもらう」を目指さないのか

最後に、この平易版そのものの立ち位置を。

この文書は、正本を**全員に理解してもらう**ためのものではない。正本の密度（argued not measured）は正本の美点であって、平らにならせば必ず何かが死ぬ。だからここでの目標は **partial resonance** ——一人が一つの角を深く受け取れば足りる、という構えだ。

そしてこれは、この論文自身の主張とぴったり重なる。§3 が言うように、価値は「相手が自分の全部を内面化すること」ではなく、「一つの角が、相手の領域で深く着地し、こちらが予期しなかった応用を返してくること」から来る。**部分的な外で、十分だ。** 銀行台帳の一例でも、二人のドライバーの死角でも、git fork の血統でも——どれか一つがあなたの中で深く着地したなら、この平易版はその仕事を終えている。
