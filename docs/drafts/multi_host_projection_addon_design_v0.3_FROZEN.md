# Multi-Host Projection: Add-on Registration Design (v0.3 draft)

## 1. Scope

This draft covers the redesign of KairosChain's projection engine responsibility boundary between the L0 core and host-specific support. It defines the invariants that must hold once host support for AI coding hosts beyond the single built-in default is delivered through independently installable add-on packages rather than through core branching. It covers: how a host profile is represented as an abstraction, how such a profile is registered and discovered, how conversion behavior is owned, how inter-profile dependencies are declared and enforced at both projection and removal time, how shared artifacts among profiles are attributed, how user-facing surfaces enumerate available hosts, how installation and removal of a profile affect previously projected artifacts, how the built-in default profile participates in the same registration mechanism as every add-on, how trust boundaries constrain profile effects, and how registration and removal events are constitutively recorded. It applies uniformly to all access paths through which projection can be initiated — including at minimum the long-running path, the standalone path, and any host-facing instruction-mode delivery.

## 2. Non-goals

This draft does not specify the projection algorithm itself, the internal representation of SkillSet artifacts, the format of any manifest, hook, context, or sub-agent file, or the user experience of installing an add-on package. It does not evaluate or rank candidate mechanisms — that evaluation, along with any naming of concrete classes, modules, files, or commands, is deferred entirely to the mechanism backlog. It does not address hosts other than the three presently known, nor does it attempt to anticipate the full space of future host conventions; the invariants are intentionally open to future hosts, guaranteeing that such hosts can be added without touching the core. It does not cover versioning or compatibility of the SkillSet package format in general, only the narrow slice relevant to host-profile registration and removal.

## 3. Invariants

INV-H1: The core recognizes exactly one unit of host-specific knowledge — the host profile — and nothing else. Any behavior that varies by target host must be expressible as a property of a profile; the core must never contain a conditional that names a host. This is the foundational invariant because every other property in this draft assumes that "host-specific" and "profile-shaped" are the same thing; if the core retained even one bypass where it reasoned about a host directly, the extraction into add-ons would be cosmetic rather than structural, and the next host added would recreate the branching this design exists to eliminate.

INV-H2: The built-in default profile is itself a bundled, pre-registered add-on package that passes through the same registration mechanism as every other profile. The core's sole privilege is that this package ships with the core and registers first; the registry is therefore never empty in a functioning installation. The default profile is removable by the same rules as any profile, with no special treatment, consistent with §4; an installation whose registry has been emptied by such removals is a non-projecting installation, and the "never empty" property is scoped to installations as shipped and to any installation that has not deliberately removed its last profile. This is required both to resolve the bootstrap ambiguity — a fresh install with no external add-ons must still function — and to maintain structural self-referentiality: if the default profile bypassed the registration mechanism it defines for all others, the mechanism's claim to generality would rest on an asymmetric exception at its foundation, and any invariant proved for "all registered profiles" would not, in fact, cover the default.

INV-H3: A profile, once registered, is discoverable identically from all access paths that can initiate projection, and no access path maintains its own separate notion of which hosts exist. The set of projection-initiating access paths includes at minimum the long-running path, the standalone path, and any host-facing instruction-mode delivery; INV-H3 and the compatibility properties in §5 apply to all of them uniformly. Registration alone never causes projection — no access path may auto-project without an explicit trigger. This matters because if any path maintained its own view of available profiles, registering an add-on through one path could silently leave another path unaware of it, producing a class of bug where behavior depends on which entry point was used rather than on the registered state; and if registration triggered projection implicitly, the operator would lose the ability to register a profile and verify its configuration before any filesystem effects occur.

INV-H4: All conversion behavior a host requires — including transformations of context, instruction, hook, or sub-agent artifacts into the shapes that host expects — is owned by the profile that registers for that host, and the core's projection machinery invokes this behavior only through the profile abstraction, never by inspecting which host is being targeted. The core must not vary behavior on host identity nor on any profile property whose value set is in one-to-one correspondence with host identities; profile properties must be behavior-bearing data that the generic machinery consumes, not host tags in disguise. This is necessary because "conversion travels with the profile" is the actual test of whether the extraction succeeded; if any transformation logic remained in the core keyed off a host identity — whether as a literal conditional or as a dispatch on a property that merely encodes host identity in another form — that logic would need to be updated every time a new host's conventions were added, which is precisely the maintenance burden this design exists to remove.

INV-H5: A profile that requires, as a precondition for its own projection, that another profile's artifacts already exist must declare that requirement as data attached to the profile itself. The precondition is satisfied only when the prerequisite profile's projected artifacts actually exist on disk and verify against their source — registry membership alone is insufficient. The core must evaluate this requirement before attempting projection, never learn of an unmet requirement only when projection partially fails. This is required because at least one anticipated add-on is defined in terms of reusing another host's already-projected artifacts rather than producing its own; without an upfront, declarative dependency evaluated against actual artifact state, the failure mode of projecting into a state where the prerequisite is absent or stale would be indistinguishable from a general projection bug, making it much harder to give the operator an actionable, pre-flight diagnosis.

INV-H6: Any user-facing functional input surface that offers a choice among hosts must derive that choice, at the moment of presentation, from the current registry contents, and must never be authored with an enumerated, fixed list of host names. Diagnostic, log, and help text may render current registry contents for informational purposes, and illustrative prose naming hosts in documentation or instruction-mode description is meta-level description, not a violation. This follows directly from the requirement that the core never know concrete host names: if a schema or prompt hardcoded a host list for functional selection purposes, that list would necessarily go stale the moment a profile is added or removed, and the core would once again be carrying host-specific knowledge — just relocated into a schema instead of a conditional, which does not satisfy the intent of the extraction.

INV-H7: Removing a profile's registration must leave no artifact in a projected target that can be attributed solely to that profile, and must not require manual cleanup by the operator beyond triggering the removal itself. Removal silently deletes only artifacts whose current content matches what projection generated; any artifact that has drifted from its generated content halts removal of that artifact and is reported for human decision — no silent destruction of hand-edited files. The record of what a profile projected is held by the core, not solely inside the add-on package, so removal remains drivable after out-of-band package deletion and is re-drivable after interruption without loss of the information needed to complete it. This is necessary because a registry that can grow but never shrink cleanly is not actually a registry in the sense this design needs, because orphaned projected files are a correctness hazard as much as a hygiene concern, and because the operator must retain control over artifacts they have intentionally modified.

INV-H8: There is exactly one shared verification routine and exactly one shared reversal routine in the core, each parameterized by profile. Add-on profiles never ship their own verification or reversal; they supply the data — artifact manifests, source references — that the shared routines consume, and the core applies the same checks and the same undo operation regardless of which profile is involved. Verification of a profile whose input is another profile's projected artifacts is defined against the true origin — the source artifacts — transitively; a composite state where each individual artifact is pointwise clean but the transitive chain is stale must be detectable. This is required because add-on-contributed profiles must not be second-class citizens relative to the built-in one; if verification or reversal could be overridden per profile, the core would no longer control the meaning of "verified" or "reversed," and a defective add-on could silently redefine those guarantees for its own artifacts.

INV-H9: A profile whose projected artifacts satisfy another registered profile's declared precondition (INV-H5) cannot be removed or reversed while that dependent profile remains registered, unless the dependent is removed first or the operator explicitly overrides with the consequence stated. An operator override is itself a recorded event under INV-H12, and the dependent profile left with an unmet precondition is thereby identifiable as such from the override record — rather than discovered only at the next transitive verification. This is required because without a reverse-dependency constraint, removing a prerequisite profile would silently invalidate a dependent profile's projection, producing a state where the dependent's artifacts reference or reuse content that no longer exists — a failure mode that would surface only at the next verification or re-projection, with no attribution to the removal that caused it.

INV-H10: Every projected artifact — including shared or region-shared documents contributed to by multiple profiles — has a defined owner set. Removal of one owner must neither destroy what remaining owners still need nor leave unowned residue. When a removal would empty an artifact's owner set, the artifact is treated as solely owned by the profile being removed, and that profile's removal rules (INV-H7, including drift halt) apply to it. For documents contributed to by multiple profiles, ownership and the INV-H7 drift baseline are defined per owner contribution, not per whole artifact: removal of one owner compares and removes only that owner's contribution, so an untouched contribution never triggers a drift halt because a co-owner's contribution changed, and reversal composes per contribution under the single shared reversal routine of INV-H8. This is required because at least one anticipated add-on contributes content to the same projected document as another profile, and without defined attribution, removal of either profile would face an undecidable choice: delete the shared artifact (breaking the other profile's projection) or leave it (violating INV-H7's no-residue requirement).

INV-H11: All filesystem effects of any profile — creation, modification, and deletion — pass through the core's single write and delete path, which enforces containment to the profile's declared output area. A profile cannot express an effect outside its declared area. A declared output area must not subsume another registered profile's declared area nor any area the projection machinery does not manage; contributions to shared documents are expressed through the owner-set discipline of INV-H10, never by declaring ownership of the whole shared area. Malformed, out-of-bounds, or subsumption-violating profile registrations are rejected at registration time, not discovered at projection time. This is required because profiles supply conversion behavior that writes into and deletes from a consumer project; without a containment boundary enforced by the core at the point of registration, a defective or adversarial profile could write outside its intended scope, and the defect would not be caught until projection had already produced the out-of-bounds artifact — too late for the operator to intervene without cleanup.

INV-H12: Registering or removing a host profile is a recorded event in the system's immutable change history. The record survives removal: "no projected artifact remains" (required by INV-H7) is distinct from "no record remains" (forbidden). This is required because registration and removal are constitutive acts — they change the system's identity in terms of which hosts it can address — and a system that can constitutively change without recording the change cannot account for its own history of self-modification, which is a precondition for the metacognitive self-referentiality this project's philosophy demands.

INV-H13: Profile identity is unique in the registry. A second registration claiming the same host identity as an already-registered profile is rejected as a conflict, deterministically, regardless of load order. This is required because if duplicate registrations were silently accepted or resolved by ordering, the effective behavior of the system would depend on an undeclared load-order assumption, and two add-on packages claiming the same host identity would produce ambiguous projection state that no verification routine could diagnose.

## 4. Layer placement

The L0 core retains: the host-profile abstraction itself (the shape a profile must conform to in order to participate); the registry through which profiles become known and discoverable; the single shared verification routine and the single shared reversal routine, both parameterized by profile and expressed solely in terms of the profile abstraction; the containment enforcement that bounds each profile's filesystem effects to its declared output area; the reverse-dependency evaluation that prevents removal of a prerequisite while a dependent remains; the owner-set tracking for shared artifacts; and the constitutive recording of registration and removal events.

The built-in default profile is a bundled add-on package that registers through the same mechanism as every external add-on. It ships with the core and registers first, ensuring that a fresh installation is functional without external packages. Its structural privilege is limited to this co-shipping; it receives no special treatment from the registration, projection, verification, or reversal machinery. This symmetry is the bootstrap resolution: the mechanism the core provides is validated by the fact that the core's own default profile is its first consumer, and any invariant that holds for "all registered profiles" holds for the default without exception.

Everything that varies by concrete host lives in an add-on SkillSet: the profile implementation itself; all conversion behavior needed to reshape context, instruction, hook, and sub-agent artifacts into that host's expected form; any declaration that the profile depends on another profile's artifacts being present; and the data (artifact manifests, source references, declared output area) that the core's shared verification and reversal routines consume.

The boundary is a property, not a file list: a change is correctly placed in the core if and only if it is expressible without reference to any concrete host, and correctly placed in an add-on if and only if it is meaningless without reference to exactly one concrete host.

## 5. Compatibility and migration

§§ 6–10 are reserved by the project template and intentionally omitted.

An installation that, prior to this redesign, already projects to all three known hosts must, after the redesign, continue to project to all three hosts with no operator-visible change in projected content, provided the corresponding add-ons are installed and registered. For each extracted host, the add-on's conversion output must be observably equivalent to the pre-extraction output for identical inputs. The migration must not require the operator to manually re-author, relocate, or hand-edit any previously projected artifact; the transition from core-hardcoded host support to add-on-registered host support is an internal restructuring, not a change in observable projection behavior.

Re-running projection after migration, with the appropriate add-ons installed and registered, must be idempotent with respect to the pre-migration projected state: it must not produce a materially different result than the pre-migration projection would have, and it must not introduce spurious differences that would cause verification to report a mismatch where none is functionally present.

Registry state — which profiles are currently registered — must persist across process restarts of the long-running path and must be reproducible on demand by the standalone path, so that an installation which has migrated and registered its add-ons does not silently revert to single-host behavior after a restart, and so that all access paths never disagree about which hosts a migrated installation currently supports.

A migrated installation whose previously projected artifacts have no corresponding registered profile must report those orphaned projections as detectable, attributable-to-no-profile state. Orphaned projections must not be silently invisible — an operator or diagnostic tool must be able to discover that projected artifacts exist for a host whose profile is not registered, and to attribute them to the absent profile by the core's own projection records (INV-H7). For artifacts projected before this redesign existed, attribution of orphans relies on projection records reconstructed or imported during migration; the migration must produce core-held records sufficient for this orphan-detection property to hold for pre-existing projections, not only for post-migration ones.

## 11. Mechanism backlog

- Host-profile interface/contract definition (the shape a profile must conform to: layout declaration, artifact conversion, dependency declaration, declared output area)
- Central profile registry, populated at add-on load time, queried by all access paths
- Bundled default profile packaged and registered through the same add-on mechanism as external profiles
- Add-on manifest field declaring inter-profile dependencies, evaluated against actual on-disk artifact state before projection begins
- Registry-driven input generation for any user-facing host-selection surface, replacing hardcoded host lists
- Shared verification routine parameterized by profile, with transitive-chain detection for dependent profiles
- Shared reversal routine parameterized by profile, with drift detection and halt-on-mismatch for hand-edited artifacts
- Owner-set tracking for shared artifacts, with removal rules that respect remaining owners
- Per-contribution ownership records for shared documents, supporting INV-H10 per-contribution granularity, per-contribution drift baseline, and per-contribution reversal
- Reverse-dependency evaluation at removal time, preventing removal of a prerequisite while a dependent remains
- Containment enforcement at registration time, rejecting profiles whose declared output area is malformed or out-of-bounds
- Constitutive recording of registration and removal events in the system's immutable change history
- Core-held projection records sufficient to drive removal and cleanup after out-of-band package deletion
- Orphaned-projection detection for migrated installations with unregistered host artifacts
- Migration-time reconstruction or import of projection records for pre-existing projected artifacts, ensuring the §5 orphan-detection property holds for pre-migration projections
- Identity-conflict rejection at registration time for duplicate host identities
- Extraction of existing Codex conversion logic into a standalone add-on SkillSet package
- Extraction of existing OpenCode conversion logic into a standalone add-on SkillSet package, carrying an explicit declared dependency on the Claude profile
- Migration validation that a pre-existing three-host installation continues to verify cleanly once both add-ons are installed and registered, with behavioral equivalence checks against pre-extraction output
